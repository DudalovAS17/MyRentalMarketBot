# 1. Аргументы колонок `mapped_column(...)`

### 1. `primary_key=True`
**[Делает поле первичным ключом таблицы]()**

Первичный ключ — это главный способ отличать одну запись от другой.

Зачем нужно:
- БД использует это поле как главный идентификатор строки.
- Значение должно быть уникальным
- Значение не должно быть NULL
- ORM через этот ключ понимает, какую конкретно запись она загружает, обновляет или удаляет.

### 2. `autoincrement=True`

**[Говорит БД автоматически увеличивать числовое значение при создании новой записи.]()**

Зачем нужно:
- При каждой новой записи БД сама выдаёт следующее число
- Обычно используется вместе с `primary_key=True`

### 3. `nullable=False`
**[Запрещает хранить NULL в колонке на уровне БД]()**

- Если поле `nullable=False`
```
if rental.created_at is not None:  # ❌ лишний шум
```

- `Optional[datetime]` + `nullable=False` — плохо, т.к. 
  * `Optional` говорит: “может быть None”, 
  * но `nullable=False` говорит: “никогда не None”

### 4. `default=...`
**[Задаёт значение по умолчанию на стороне Python / ORM.]()**

Зачем нужно:
- Если ты не передал значение вручную, SQLAlchemy подставит его сам.
- Работает до отправки данных в БД.
- Часто используется для enum-статусов, счётчиков, булевых флагов.

### 5. `server_default=...`
**[Задаёт значение по умолчанию на стороне БД.]()**

- Значение создаёт сама база данных.
- Полезно для технических дат вроде created_at.
- Работает даже если запись создана не через Python-код, а напрямую в БД.

Почему это лучше для дат создания:
```
✅ server_default=func.now()
❌ default=lambda: datetime.now(timezone.utc)
```
Потому что время фиксирует сама БД, а не Python-приложение.


### 6. `onupdate=...`
**[Автоматически обновляет значение при изменении записи.]()**

Зачем нужно:
- Используется для `updated_at`.
- Позволяет автоматически фиксировать время последнего обновления.
- Обычно ставится вместе с `server_default`.

Хороший вариант:
```
onupdate=func.now()
```
Плохой вариант:
```
onupdate=lambda: datetime.now(timezone.utc)
```

### 7. `index=True`
**[Создаёт индекс для колонки.]()**

Зачем нужно:
- Ускоряет поиск и фильтрацию по этому полю

Например, если часто ищешь записи по `parent_id`, индекс помогает БД быстрее находить нужные строки.

Важно: `index=True` = быстрый поиск, но лучше использовать явный `Index(...)`, чтобы не плодить мусор в БД.


### 8. `unique=True`
**[Запрещает повторяющиеся значения в колонке.]()**

Зачем нужно:
- Делает поле уникальным
- Например, один Telegram ID не должен принадлежать двум пользователям
- БД сама не даст вставить дубликат

Простая мысль:
`unique=True` = “такое значение может быть только одно в таблице”.

# 2. Аргументы типов данных

### 1. `DateTime(timezone=True)`
**[Создаёт колонку с датой и временем с учётом timezone.]()**

Зачем нужно:
- Позволяет хранить timezone-aware datetime
- Лучше использовать aware UTC-время, чтобы не ловить ошибки из-за часовых поясов

Простая мысль: `timezone=True` = “эта дата знает свой часовой пояс”

### 2. `Numeric(12, 2)`
**[Тип для точных чисел, особенно денег.]()**

Зачем нужно:
- Используется вместо `float`, чтобы избежать ошибок округления
- Подходит для цены, депозита, суммы
- `12` — общее количество цифр
- `2` — количество цифр после запятой

“Точное денежное число с двумя знаками после запятой”: `9999999999.99`

### 3. `BigInteger`
**[Целое число большого размера.]()**

Зачем нужно:
- Используется там, где обычного `Integer` может не хватить
- актуально для `Telegram ID`, потому что они могут быть большими

“Большое целое число”

### 4. JSON
**[Тип для хранения JSON-структуры.]()**

Зачем нужно:
- Можно хранить словари, списки, вложенные данные
- Подходит для координат, payload, дополнительных деталей события
- Удобно, когда структура данных гибкая

“Храним структурированные данные прямо в колонке”: `{"lat": 55.7, "lng": 49.1}`

### 5. SAEnum(SomeStatus, name="...")
**[SQLAlchemy Enum для хранения статусов.]()**

Зачем нужно:
- Ограничивает значение заранее заданным набором вариантов
- Например: `PENDING, ACTIVE, REQUESTED, OPEN`
- Делает модель безопаснее, чем обычная строка
- `name="..."` задаёт имя enum-типа в БД

Примеры:
- `SAEnum(ItemStatus, name="item_status")`
- `SAEnum(RentalStatus, name="rental_status")`
- `SAEnum(AccountStatus, name="account_status")`
- `SAEnum(SupportTicketStatus, name="support_ticket_status")`

`SAEnum` = “статус может быть только из разрешённого списка”

# 3. `ForeignKey("...", ondelete=...)` 

**[Создаёт внешний ключ.]()**

Зачем нужно:
- [Связывает одну таблицу с другой]()
- Говорит БД: значение в этой колонке должно ссылаться на существующую запись
- Защищает от “битых ссылок”

Пример: `ForeignKey("users.id")`
  * это поле ссылается на `id` в таблице `users`

`ondelete=`
```
- "CASCADE" - При удалении родителя автоматически удаляет связанные дочерние записи. (временные данные)
- "RESTRICT" - Запрещает удалить родителя, если на него есть ссылки = есть история. (История / финансы / сделки)
- "SET NULL" - При удалении связанной записи ставит NULL в ссылку. (мягкие связи)
- "NO ACTION" - Поведение зависит от конкретной БД и момента проверки ограничения.
```

Пример: `ForeignKey("categories.id", ondelete="CASCADE")` 
  * если удалить родительскую категорию, БД удалит и дочерние категории
  * делает дерево категорий жёстко зависимым от родителя
  * подкатегория не считается самостоятельной сущностью без родителя

# 4. Отношения: Аргументы `relationship(...)`

### 1. `back_populates="..."`
**[Связывает две стороны одной ORM-связи.]()**

Зачем нужно:
- Говорит SQLAlchemy, что два relationship относятся к одной и той же связи. Склеивает их в одну согласованную ORM-связь
- Например, одна сторона — “мой родитель” (связь “вверх”), другая — “мои дети” (связь “вниз”)
- ORM начинает синхронизировать эти отношения между объектами

Пример смысла:
- `parent <-> subcategories`
- `owner <-> items`
- `item <-> rentals`

“Обратные стороны одной связи”


### 2. `remote_side="..."`
**[Используется в self-referential relationship, когда модель ссылается сама на себя.]()**

Зачем нужно:
- Без него SQLAlchemy не понимает, где родитель, а где потомок
- Особенно важно для дерева категорий
- говорит ORM, что именно `Category.id` является “удалённой” стороной связи

`remote_side="Category.id"`

Пример ситуации: `Category.parent_id -> Category.id`. 
Таблица одна и та же, поэтому ORM нужно объяснить направление связи.

“В связи модели самой с собой вот это сторона родителя”

### 3. `cascade="all, delete-orphan"`
**[ORM-каскад для дочерних объектов.]()**

Состоит из двух важных частей:

A) `cascade="all"`

**[Говорит ORM применять основные операции к дочерним объектам вместе с родителем.]()**

Зачем нужно:
- Если родитель сохраняется, обновляется или удаляется, ORM учитывает и связанные дочерние объекты.
- Удобно для настоящих parent-child связей.

“Основные ORM-операции распространяются на детей”

B) `delete-orphan`

**[Удаляет дочерний объект, если он потерял родителя.]()**

Зачем нужно:
- Если фото больше не принадлежит вещи — оно удаляется
- Если дочерняя категория оторвалась от родителя — ORM может считать её сиротой
- Подходит только для сущностей, которые не должны жить самостоятельно

Опасный момент: В self-referential связях delete-orphan требует осторожности и 
часто должен идти вместе с `single_parent=True`

“Объект без родителя считается сиротой и удаляется”

> Пример:
>
> Убрал `cascade="all, delete-orphan`, почему: у `Rental.item_id` стоит 
> `ForeignKey("items.id", ondelete="RESTRICT")` = вещь с историей аренды удалять нельзя.
> Следовательно, удаление `Item` не должно автоматически удалять `Rental`.
>
> А `cascade="all, delete-orphan"` здесь как раз означал бы:
> удалил `Item` → ORM начинает удалять связанные `Rental`.
> Это противоречит смыслу модели аренды как исторически значимой сущности.

### 4. `single_parent=True`
**[Говорит ORM, что дочерний объект может иметь только одного родителя.]()**

Зачем нужно:
- Нужно при delete-orphan (и объект должен иметь только одного владельца)
- Защищает от ситуации, когда один child привязан сразу к нескольким parent
- Делает владение объектом однозначным

Пример смысла:
- одно фото принадлежит одной вещи
- одна вещь принадлежит одному владельцу
- одна подкатегория имеет одного родителя

### 5. `passive_deletes=True`
**[Говорит ORM не удалять дочерние записи вручную, а доверить это базе данных.]()**

Зачем нужно:
- Если в БД уже есть ON DELETE CASCADE, ORM не обязана загружать всех детей и удалять их по одному
- БД сама удаляет все связанные photo
- Это уменьшает количество лишних SQL-запросов
- Особенно полезно, если у родителя много дочерних записей
- Это снижает лишнюю ORM-активность и лучше согласует поведение с FK-правилами БД

“ORM не трогает детей, БД сама разберётся по FK-правилам”

### 6. `foreign_keys=[...]`
**[Явно указывает, через какой внешний ключ строить связь.]()**

Зачем нужно:
- Нужно, когда между двумя таблицами есть несколько FK
- Например, аренда связана с пользователем как owner и как renter
- Без `foreign_keys` SQLAlchemy не поймёт, какую колонку использовать

Пример смысла:
- `foreign_keys=[owner_id]`
- `foreign_keys=[renter_id]`

“Строй эту relationship именно через вот этот FK”

### 7. `lazy="joined"`
**[Стратегия загрузки relationship через JOIN.]()**

Зачем нужно:
- Связанные данные загружаются сразу одним SQL-запросом через join
- Удобно, если почти всегда нужна связанная сущность
- Может быть тяжёлым, если связей много

“Загрузи связанные данные сразу через JOIN”

### 8. `lazy="selectin"`
**[Стратегия загрузки relationship отдельным дополнительным запросом через IN.]()**

Зачем нужно:
- Хорошо подходит для списков дочерних объектов
- Позволяет избежать N+1 проблемы
- Часто лучше, чем joined, для коллекций

“Загрузи связанные объекты отдельным умным запросом”

# 5. Отношения: Аргументы ограничений и индексов

`__table_args__` - **[Место, где задаются ограничения и индексы таблицы.]()**

Зачем нужно:
- Можно явно описать `Index`
- Можно добавить `UniqueConstraint`
- Можно добавить `CheckConstraint`
- Удобно держать сложные правила таблицы в одном месте

“Дополнительные правила таблицы”

### 1. `UniqueConstraint(...)`
**[Создаёт уникальное ограничение.]()**

Зачем нужно:
- Запрещает повторение определённой комбинации значений
- Может работать не только по одному полю, но и по нескольким
- Например, имя категории уникально только внутри одного родителя
- Одинаковые имена допустимы в разных ветках дерева

Пример из файла:
`UniqueConstraint("parent_id", "name", name="uq_categories_parent_id_name")`

Смысл: `parent_id + name` - должны быть уникальными вместе.

“Такая комбинация значений не должна повторяться”

### 2. `CheckConstraint(...)`
**[Создаёт проверку на уровне БД.]()**

Зачем нужно:
- БД сама проверяет условие перед вставкой или обновлением
- Это последняя линия обороны от плохих данных
- Например, можно запретить категории быть родителем самой себе
- Значит модель БД запрещает самый грубый случай, но не решает полную задачу ацикличности дерева

Пример из файла:

`CheckConstraint("parent_id IS NULL OR parent_id <> id", name="ck_categories_no_self_parent")`

Важно:

Такой `check` защищает только от простого случая: `A -> A` (“сам себе родитель”).
Но не защищает от длинного цикла: `A -> B -> C -> A`

“БД проверяет это условие сама”

### 3. `Index(...)`
**[Создаёт индекс с явным именем.]()**

Зачем нужно:
- Ускоряет частые запросы (например, для быстрого получения категорий по parent_id)
- Например, получить всех детей конкретного родителя
- Лучше, чем `index=True`, если ты хочешь контролировать имена индексов
- Без индекса выборка подкатегорий по `parent_id` будет хуже масштабироваться

Пример: `Index("ix_categories_parent_id", "parent_id")`

“Ускорь поиск по этому полю/полям”

### 4. `name="..."`
**[Явно задаёт имя ограничения, индекса или enum-типа.]()**

Зачем нужно:
- Удобно для Alembic-миграций
- Удобно читать схему БД / отлаживать ошибки
- Делает структуру БД предсказуемой

“Дай этому правилу нормальное понятное имя”

# 6. Naming Convention
**[Единый шаблон имён для ограничений и индексов.]()**

Зачем нужно:
- Alembic лучше понимает изменения схемы
- Имена PK/FK/UQ/CK/IX становятся единообразными
- Схему проще читать
- Меньше хаоса в миграциях

```
NAMING_CONVENTION = {
    "ix": "ix_%(table_name)s_%(column_0_N_name)s",
    "uq": "uq_%(table_name)s_%(column_0_N_name)s",
    "ck": "ck_%(table_name)s_%(constraint_name)s",
    "fk": "fk_%(table_name)s_%(column_0_N_name)s_%(referred_table_name)s",
    "pk": "pk_%(table_name)s",
}
```

```
ix	индекс
uq	unique constraint
ck	check constraint
fk	foreign key
pk	primary key
```

“Единый стиль имён для всех правил БД”

# Примеры:

>Главный принцип:
>- **родитель** = [сущность, от которой зависит другая]()
>- **child** = [живёт “под” родителем]()
>- **историческая связь** = [удалять родителя нельзя, пока есть история]()
>- **мягкая ссылка** = [связь есть, но при удалении можно обнулить]()

В моделях это видно по 
- `relationship(...)`
- `cascade=...`
- и особенно по `ForeignKey(..., ondelete=...)`

## [родитель → ребенок]()

### A. `User -> Item` = родитель → дочерняя сущность

Почему:
- в `Item.user_id` -> `users.id` стоит `ondelete="CASCADE"`
- в `User.items` стоит `cascade="all, delete-orphan"` и `single_parent=True`

**Смысл:**
- вещь принадлежит пользователю
- без владельца эта вещь в текущей модели не должна жить

```
User (родитель)
  └─ Item (дочерняя сущность)
```

### B. `Item -> Photo` = родитель → дочерняя сущность

Почему:
- `Photo.item_id` -> `items.id` с `ondelete="CASCADE"`
- `Item.item_photos` с `cascade="all, delete-orphan"` и `single_parent=True`

**Смысл**: фото не существует отдельно от вещи.

```
Item (родитель)
  └─ Photo (дочерняя сущность)
```

### C. `Category -> Category` = дерево, self-referential relationship

Почему:
- `Category.parent_id -> categories.id`
- `parent` и `subcategories` связывают категорию саму с собой

**Смысл**: одна категория может быть родителем для подкатегорий.

```
Category (родитель)
  └─ Category (подкатегория)
```

## [История / нельзя удалять (не “родитель-ребёнок”)]()

### D. `Item -> Rental` - историческая связь.

Почему:
- `Rental.item_id` -> `items.id` с `ondelete="RESTRICT"`
- у `Item.rentals` нет `delete-orphan` каскада

**Смысл**: 
- аренда — это история сделки
- нельзя удалить `item`, если по ней есть история аренды

```
Item
  └─ Rental
```

### E. `User -> Rental` - историческая связь.

Почему:
- `Rental.renter_id` и `Rental.owner_id` обе с `ondelete="RESTRICT"`

**Смысл**: 
- нельзя удалять пользователя, если он участвовал в арендах
- аренда хранит историю владельца и арендатора

```
User (owner)  ─┐
               ├─ Rental (история сделки)
User (renter) ─┘
```

### F. `Rental -> Review` - это ближе к дочерней сущности сделки.

Почему:
- `Review.rental_id` -> `rentals.id` с `ondelete="CASCADE"`
- `Rental.reviews` с `cascade="all, delete-orphan"`

Смысл:
- отзыв живёт внутри контекста конкретной аренды
- если аренды нет, отзыв теряет смысл

```
Rental (родитель)
  └─ Review (дочерняя сущность сделки)
```

Но при этом `Review.reviewer_id` и `reviewee_id` на `User` идут через `RESTRICT`, 
то есть пользователей удалять нельзя, если отзывы уже существуют.


### G. `User -> SupportTicket` - это исторически важная, но не “delete-orphan child” связь.

Почему:
- `SupportTicket.user_id` -> `users.id` с `ondelete="RESTRICT"`
- тикет — часть истории поддержки.

```
User
  └─ SupportTicket   (история поддержки, не удалять каскадом)
```

## [“мягкая ссылка”]()

### H. `Item -> subcategory` - мягкая ссылка

Почему:
`subcategory_id` -> `categories.id` с `ondelete="SET NULL"`

Смысл:
- если подкатегория исчезнет, вещь не обязана исчезнуть
- можно просто обнулить subcategory_id

```
Item
  └─ subcategory_id -> Category
       if deleted -> NULL
```

### I. `Item -> moderated_by_admin_id` - мягкая ссылка

Почему:
- `moderated_by_admin_id` -> `users.id` с `ondelete="SET NULL"`

Смысл:
- запись о модерации может остаться, даже если конкретный admin-user исчезнет из ссылки.

### J. `User -> banned_by_admin_id`

То же самое:
- `banned_by_admin_id` -> `users.id` с `SET NULL`

## [Вообще нет FK-родителя]()

### K. `AdminAction` - стоит особняком.

Почему:
- у него `admin_id` хранится просто как `BigInteger`;
- это не FK на `users.id`

Смысл:
- audit log не зависит от ORM-связи с `User`;
- он хранит внешний идентификатор админа как значение.

```
AdminAction
  └─ admin_id/admin_tg_id   (значение, а не ORM-родительская связь)
```

## [Карта именно “по типам связей”]()

### Настоящие parent-child
- `User -> Item`
- `Item -> Photo`
- `Category -> subcategories`
- `Rental -> Review`

### История / RESTRICT
- `Item -> Rental`
- `User -> Rental`
- `User -> Review`
- `User -> SupportTicket`

### Мягкие ссылки / SET NULL
- `Item -> subcategory`
- `Item -> moderated_by_admin_id`
- `User -> banned_by_admin_id`

### Без FK-связи
- `AdminAction.admin_id/admin_tg_id`

## [Простым языком]()

Когда смотришь на новую связь в модели, задавай 3 вопроса:

### 1. Может ли сущность жить без родителя?

Если нет, это parent-child.

Примеры:
- `Photo` без `Item` — нет
- `Item` без `User` в твоей модели — нет

### 2. Это история, которую нельзя терять?

Если да, скорее всего `RESTRICT`.

Примеры:
- `Rental`
- `Review`
- `SupportTicket`

### 3. Это просто дополнительная ссылка?

Если да, возможен SET NULL.

Примеры:
- `subcategory_id`
- `moderated_by_admin_id`
