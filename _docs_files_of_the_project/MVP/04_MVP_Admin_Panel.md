# MVP: Админка

## 1. Назначение блока

Админка превращает бота из каталога в рабочий бизнес-инструмент.

Главная задача админки:

```text
менеджер видит заявки
→ берёт их в работу
→ подтверждает / отклоняет / завершает
→ клиент получает статус
```

Для MVP админка должна быть не полной CRM, а удобным рабочим местом менеджера внутри Telegram.

---

## 2. Что уже есть в проекте

Уже есть:

```text
отдельный admin router
admin middleware / защита доступа
админ-меню
список заявок
карточка заявки
смена статусов
take_in_progress
confirm_rental
reject_rental
admin_cancel_rental
complete_rental
уведомление клиента после смены статуса
AdminAction / audit log
работа с клиентами
базовая поддержка
часть управления товарами
```

Это уже близко к MVP.

---

## 3. MVP-минимум админки

Админка должна содержать:

```text
🆕 Новые заявки
📋 Все заявки
💬 Поддержка
📦 Товары / каталог
👥 Клиенты
```

Для первичного MVP главное — заявки и поддержка.

---

## 4. MVP по заявкам

В карточке заявки менеджер должен видеть:

```text
номер заявки
статус
товар
количество
срок
доставка
адрес
стоимость, если рассчитана
имя клиента
телефон
username / telegram id
комментарий клиента
комментарий менеджера
дата создания
```

Кнопки:

```text
✅ Взять в работу
✅ Подтвердить
❌ Отклонить
🚫 Отменить
🏁 Завершить
📞 Контакт клиента
📝 Комментарий менеджера
⬅️ Назад
```

Кнопки должны зависеть от текущего статуса.

---

## 5. Что нужно слегка поправить

### 5.1. Разделить “Новые заявки” и “Все заявки”

Сейчас может быть список последних заявок. Для MVP лучше явно разделить:

```text
🆕 Новые заявки = только REQUESTED
📋 Все заявки = все статусы
```

Минимальный сервисный метод:

```text
list_rentals_by_status(REQUESTED)
```

### 5.2. Добавить контакт клиента

В карточке заявки нужна кнопка:

```text
📞 Контакт клиента
```

Она показывает:

```text
телефон
имя
username
telegram id
```

Если есть username, можно показать ссылку вида:

```text
https://t.me/username
```

Важно: бот не всегда может первым написать пользователю вне текущего диалога, но менеджер должен видеть контакты.

### 5.3. Отдельный комментарий менеджера

Не смешивать:

```text
manager_comment
reject_reason
cancel_reason
```

Для MVP достаточно добавить отдельную кнопку:

```text
📝 Комментарий менеджера
```

Она открывает FSM ввода/обновления `manager_comment`.

---

## 6. MVP по товарам в админке

Полноценное создание и редактирование товаров не обязательно для первого MVP.

Достаточно:

```text
посмотреть товары
скрыть / показать товар
изменить статус
```

Если создание товара уже частично есть — оставить, но не развивать выше заявки.

Терминология:

| Старое | Новое |
|---|---|
| создать объявление | добавить товар |
| модерация объявлений | управление товарами |
| объявление | товар |

---

## 7. MVP по клиентам

Для MVP достаточно:

```text
список клиентов
карточка клиента
заявки клиента
контакты клиента
```

Бан/разбан можно оставить, если уже есть, но это не центральная функция MVP.

---

## 8. Что НЕ входит в MVP админки

Пока не делать:

```text
полную CRM
роли с очень тонкими правами
дашборды аналитики
экспорт Excel
массовые рассылки
сложную модерацию
календарь занятости оборудования
```

Audit log уже есть — это плюс, его можно оставить.

---

## 9. Критерии готовности

Админка готова для MVP, если менеджер может:

```text
открыть новые заявки
открыть любую заявку
увидеть все важные данные
взять заявку в работу
подтвердить заявку
отклонить заявку
завершить заявку
посмотреть контакты клиента
добавить комментарий
ответить на обращение поддержки
```

---

## 10. Приоритет

Статус: рабочая база уже есть.

Приоритет доработки: высокий после формы заявки.

Главная цель — сделать админку mini-CRM для обработки заявок, а не просто набором кнопок статуса.
