# MVP: Деплой и запуск

## 1. Назначение блока

Деплой и запуск нужны, чтобы проект можно было не только разрабатывать, но и показать, передать, запустить на сервере или оформить в портфолио.

Для MVP цель простая:

```text
проект можно поднять по инструкции без ручного гадания
```

---

## 2. Что уже есть в проекте

По текущему направлению проекта уже обсуждались:

```text
.env-конфигурация
DEBUG polling
PROD webhook позже
логирование
БД
seed/import-скрипты
слоистая архитектура
```

Этого достаточно, чтобы оформить минимальный запуск.

---

## 3. MVP-минимум запуска

Нужно иметь:

```text
.env.example
requirements.txt / pyproject.toml
инструкцию установки
команду запуска бота
команду создания/обновления БД
команду seed demo-каталога
папку storage для фото
логирование
README с основными сценариями
```

---

## 4. Режимы запуска

### 4.1. MVP / разработка

Для разработки достаточно polling:

```text
DEBUG=True
bot запускается через polling
```

Это проще и быстрее.

### 4.2. Production позже

Для production можно подготовить, но не обязательно завершать в MVP:

```text
webhook
FastAPI/aiohttp endpoint
Docker
reverse proxy
HTTPS
```

Главное — не блокировать MVP из-за webhook.

---

## 5. Минимальный `.env.example`

```text
BOT_TOKEN=
ADMIN_IDS=
DATABASE_URL=
DEBUG=True
LOG_LEVEL=INFO
STORAGE_PATH=storage
```

Опционально:

```text
MANAGER_CHAT_ID=
SUPPORT_CHAT_ID=
WEBHOOK_URL=
WEBHOOK_SECRET=
```

---

## 6. README для MVP

В README должны быть разделы:

```text
что это за бот
основной пользовательский сценарий
основной админский сценарий
как установить зависимости
как настроить .env
как создать БД
как загрузить seed-данные
как запустить бота
какие команды есть
какие роли есть
```

---

## 7. Что нужно подправить

1. Проверить, что токены и секреты не лежат в коде.
2. Добавить `.env.example` без реальных секретов.
3. Добавить понятную команду seed demo-каталога.
4. Проверить, что проект стартует на чистой базе.
5. Проверить, что storage-файлы не ломают запуск.
6. Добавить короткий сценарий проверки после запуска.

---

## 8. Что НЕ входит в MVP

Пока не обязательно:

```text
полный production webhook
CI/CD
автоматический деплой
мониторинг
backup-система
Docker Swarm/Kubernetes
сложная observability
```

Docker можно добавить, если он уже есть или не тормозит разработку, но он не должен мешать завершению бизнес-потока.

---

## 9. Критерии готовности

Блок готов, если:

```text
новый разработчик может открыть README
→ создать .env
→ установить зависимости
→ создать БД
→ загрузить demo-каталог
→ запустить бота
→ пройти сценарий каталог → заявка → админка
```

---

## 10. Приоритет

Статус: нужно оформить перед упаковкой.

Приоритет: средний сейчас, высокий перед демонстрацией/портфолио.

Главная цель — воспроизводимый запуск, а не идеальный production-deploy.
