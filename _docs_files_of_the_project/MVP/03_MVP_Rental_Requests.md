# MVP: Заявки на аренду

## 1. Назначение блока

Заявка — главное бизнес-событие RentalMarketBot.

Цель заявки:

```text
клиент выбрал товар
→ оставил структурированную заявку
→ менеджер получил понятную карточку
→ компания не потеряла обращение
```

В текущем проекте модель `Rental` по смыслу уже является заявкой клиента на аренду. Переименовывать её сейчас не обязательно.

---

## 2. Что уже есть в проекте

Уже есть:

```text
создание заявки из карточки товара
Rental / rentals
статус REQUESTED по умолчанию
статусы обработки заявки
клиентские “мои заявки”
уведомление клиента после создания
уведомление менеджеров о новой заявке
админский список заявок
админская обработка статусов
```

В модели уже заложено много нужных полей:

```text
item_id
user_id
rental_period_text
total_price / final_price
status
quantity
delivery_needed
delivery_address
client_name
client_phone
client_comment
manager_comment
assigned_manager_id
timestamps
```

То есть backend-база хорошая. Главный долг — клиентский flow создания заявки.

---

## 3. MVP-минимум заявки

Заявка должна собирать не свободный текст, а структуру:

```text
1. Товар
2. Количество
3. Срок аренды
4. Нужна ли доставка
5. Адрес доставки, если нужна
6. Имя клиента
7. Телефон клиента
8. Комментарий
9. Подтверждение
```

После подтверждения создаётся запись `Rental`.

---

## 4. Целевой FSM-flow

Минимальный сценарий:

```text
start_rental_request
↓
quantity
↓
rental_period
↓
delivery_needed
↓
delivery_address, если нужна доставка
↓
client_name
↓
client_phone
↓
client_comment
↓
confirmation
↓
create Rental
↓
notify client
↓
notify admins
```

На каждом шаге нужны кнопки:

```text
⬅️ Назад - пока не нужно!
❌ Отменить - это нужно!
```

---

## 5. Что нужно поправить в текущем проекте

### 5.1. Вернуть quantity

Если `quantity` сейчас закомментирован в `RentalCreate`, его нужно вернуть.

Минимально:

```text
quantity: int = 1
```

Проверка:

```text
quantity >= 1
quantity <= item.available_quantity, если наличие известно
```

### 5.2. Добавить доставку

В заявке должны заполняться:

```text
delivery_needed: bool
delivery_address: Optional[str]
```

Если доставка не нужна:

```text
delivery_needed = False
delivery_address = None
```

### 5.3. Подтверждать имя и телефон

Если имя/телефон уже есть в профиле, их можно предложить по умолчанию:

```text
Использовать телефон из профиля?
```

Но для MVP важно дать пользователю возможность исправить контакт перед отправкой заявки.

### 5.4. Сделать экран подтверждения

Перед созданием заявки показать:

```text
Проверьте заявку:

Товар: ...
Количество: ...
Срок: ...
Доставка: ...
Адрес: ...
Имя: ...
Телефон: ...
Комментарий: ...

✅ Отправить заявку
✏️ Изменить - пока не нужно!
❌ Отменить
```

Логика "Изменить" пока закомменчена и не будет реализована!

## 6. Итоговая карточка заявки для менеджера

После создания заявки менеджер должен получить структурированное сообщение:

```text
🆕 Новая заявка #42

Товар: Виброплита Champion PC9045F
Количество: 1
Срок: 2–3 суток
Доставка: нужна
Адрес: ...

Клиент: Алексей
Телефон: +7...
Telegram: @username

Комментарий:
Нужна на выходные.

Статус: новая
```

Кнопки:

```text
✅ Взять в работу
✅ Подтвердить
❌ Отклонить
📞 Контакт клиента
📝 Комментарий
```

---

## 7. Что НЕ входит в MVP заявки

Пока не делать:

```text
онлайн-оплату
залог внутри бота
корзину
несколько товаров в одной заявке
автоматический расчёт сложной цены
договор аренды
интеграцию с CRM
календарь занятости оборудования
```

Для аренды оборудования нормально, что финальную цену подтверждает менеджер.

---

## 8. Критерии готовности

Заявки готовы для MVP, если:

```text
клиент может оформить заявку без свободной путаницы
менеджер получает все ключевые поля
заявка сохраняется в БД
заявка появляется в админке
клиент видит её в “Мои заявки”
менеджер может сменить статус
клиент получает уведомление о статусе
```
