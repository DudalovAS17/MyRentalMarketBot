# MVP: Пользователь / профиль

## 1. Назначение блока

Пользователь в RentalMarketBot — это не арендодатель и не продавец. Это клиент компании, который смотрит каталог, оставляет заявки и при необходимости обращается в поддержку.

Главная задача профиля в MVP — не создавать “личный кабинет”, а хранить данные, необходимые для заявки и связи с клиентом.

```text
User = клиент
Профиль = контактные данные + история заявок
```

---

## 2. Что уже есть в проекте

По текущему состоянию проекта уже есть:

```text
User-модель
создание/учёт пользователя при входе
экран профиля
клиентские “мои заявки”
использование имени и телефона при создании заявки
админская карточка пользователя
бан/разбан пользователя
```

Также в проекте уже встречаются дополнительные элементы вроде статистики и достижений. Для MVP аренды оборудования это вторично.

---

## 3. MVP-минимум

Для MVP профиль должен закрывать только практические задачи:

1. Создать пользователя при `/start`, если его ещё нет.
2. Хранить Telegram ID, username, имя.
3. Хранить телефон клиента, если он был введён или подтверждён в заявке.
4. Показывать клиенту его заявки.
5. Позволять обновить имя и телефон.
6. Не давать заблокированному пользователю создавать заявки и тикеты.
7. Давать админу посмотреть клиента и его заявки.

---

## 4. Минимальные данные

```text
User
- id
- telegram_id
- username
- full_name / name
- phone
- account_status
- created_at
- updated_at
```

Если в текущем коде поля называются иначе, переименовывать ради красоты не нужно. Главное — чтобы профиль мог отдать имя и телефон в заявку.

---

## 5. Что нужно подправить

### 5.1. Профиль должен работать на заявку

При оформлении заявки клиент должен видеть:

```text
Имя: ...
Телефон: ...
```

И иметь возможность:

```text
✅ Подтвердить
✏️ Изменить
```

То есть профиль не должен быть отдельной “игрушкой”; он должен помогать быстро оформить заявку.

### 5.2. Терминология

В клиентском профиле использовать:

```text
Мои заявки
Контактные данные
История обращений - пока не реализуем!
```

Не использовать:

```text
мои сделки
мои объявления
моя статистика арендодателя
```

### 5.3. Статистика и достижения

Если они уже есть — не обязательно удалять. Но в MVP их лучше скрыть или не развивать, чтобы не смешивать аренду оборудования с игровым/социальным профилем.

---

## 6. Что НЕ входит в MVP

Пока не делать:

```text
рейтинги клиентов
достижения
сложную статистику
баллы/уровни
избранное как часть профиля
персональные рекомендации
```

---

## 7. Критерии готовности

Блок готов, если:

```text
новый пользователь запускает бота
→ сохраняется в базе
→ может открыть профиль
→ может увидеть/изменить имя и телефон
→ может открыть “Мои заявки”
→ эти данные используются при оформлении заявки
```

---

## 8. Приоритет

Статус: частично готов.

Приоритет: средний.

Главное — связать профиль с новой полноценной FSM-заявкой, а не расширять профиль дополнительными функциями.
