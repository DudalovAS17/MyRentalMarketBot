# MVP: Статусы и переходы

## 1. Назначение блока

Статусы заявки нужны, чтобы менеджер и клиент одинаково понимали, что происходит с запросом на аренду.

Для MVP статусная модель должна быть простой, но строгой.

```text
новая заявка → в работе → подтверждена / отклонена → завершена
```

---

## 2. Что уже есть в проекте

В текущем проекте уже реализована сильная база:

```text
RentalStatus
REQUESTED
IN_PROGRESS
CONFIRMED
REJECTED
COMPLETED
CANCELLED_BY_CLIENT
CANCELLED_BY_ADMIN
терминальные статусы
открытые статусы
разрешённые переходы
timestamp-поля переходов
уведомления о смене статуса
AdminAction/audit-log
```

Это соответствует целевой модели MVP. Сейчас важно не усложнять, а формализовать правила.

---

## 3. MVP-статусы

Использовать текущий набор:

| Статус | Смысл |
|---|---|
| `REQUESTED` | новая заявка |
| `IN_PROGRESS` | менеджер взял в работу |
| `CONFIRMED` | заявка подтверждена |
| `REJECTED` | заявка отклонена |
| `COMPLETED` | аренда/заявка завершена |
| `CANCELLED_BY_CLIENT` | клиент отменил заявку |
| `CANCELLED_BY_ADMIN` | менеджер отменил заявку |

---

## 4. Минимальные переходы

```text
REQUESTED → IN_PROGRESS
REQUESTED → CONFIRMED
REQUESTED → REJECTED
REQUESTED → CANCELLED_BY_ADMIN
REQUESTED → CANCELLED_BY_CLIENT

IN_PROGRESS → CONFIRMED
IN_PROGRESS → REJECTED
IN_PROGRESS → CANCELLED_BY_ADMIN
IN_PROGRESS → CANCELLED_BY_CLIENT

CONFIRMED → COMPLETED
CONFIRMED → CANCELLED_BY_ADMIN
CONFIRMED → CANCELLED_BY_CLIENT
```

Терминальные статусы:

```text
REJECTED
COMPLETED
CANCELLED_BY_CLIENT
CANCELLED_BY_ADMIN
```

Из терминальных статусов переходы в MVP не нужны.

---

## 5. Кто может менять статус

| Действие | Кто |
|---|---|
| создать заявку | клиент |
| взять в работу | manager/admin/owner |
| подтвердить | manager/admin/owner |
| отклонить | manager/admin/owner |
| завершить | manager/admin/owner |
| отменить клиентом | клиент |
| отменить админом | manager/admin/owner |

---

## 6. Уведомления при переходах

Минимально:

```text
REQUESTED создана → уведомить менеджеров
IN_PROGRESS → уведомить клиента опционально
CONFIRMED → уведомить клиента обязательно
REJECTED → уведомить клиента обязательно
COMPLETED → уведомить клиента опционально
CANCELLED_BY_CLIENT → уведомить менеджеров
CANCELLED_BY_ADMIN → уведомить клиента
```

---

## 7. Что нужно подправить

1. Проверить, что все переходы идут через сервис, а не напрямую из handler.
2. Запретить переходы из терминальных статусов.
3. Разделить комментарии:

```text
manager_comment
reject_reason
cancel_reason
```

4. На каждый важный переход писать `AdminAction`.
5. В карточке заявки показывать статус понятным русским текстом.

---

## 8. Что НЕ входит в MVP

Пока не делать:

```text
DISPUTED
ACTIVE как отдельная аренда
частичное выполнение
продление аренды
штрафы
залоговые статусы
```

---

## 9. Критерии готовности

Блок готов, если:

```text
заявка всегда имеет понятный статус
нельзя сделать неправильный переход
клиент видит актуальный статус
админ видит доступные действия по статусу
смена статуса логируется и отправляет нужное уведомление
```

---

## 10. Приоритет

Статус: в основном готов.

Приоритет: высокий для проверки и формализации.

Не расширять статусы, пока не завершён MVP-поток заявки.
