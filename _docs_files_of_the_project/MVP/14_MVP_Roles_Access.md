# MVP: Роли и доступы

## 1. Назначение блока

Роли нужны, чтобы обычный клиент не мог попасть в админку, а менеджеры видели только то, что им нужно для работы.

Для MVP не нужна сложная корпоративная RBAC-система. Нужны понятные роли и строгая защита админских действий.

---

## 2. Что уже есть в проекте

В проекте уже есть:

```text
Admin-модель / админские сущности
админский router
middleware для защиты админки
роли owner/admin/manager по целевой концепции
админские действия и audit-log
бан/разбан пользователей
```

Это хорошая база. Нужно только формализовать минимальные права.

---

## 3. MVP-роли

```text
client — обычный пользователь
manager — менеджер заявок и поддержки
admin — администратор каталога и заявок
owner — владелец системы
```

---

## 4. Минимальные права

| Возможность | client | manager | admin | owner |
|---|---:|---:|---:|---:|
| Смотреть каталог | ✅ | ✅ | ✅ | ✅ |
| Создавать заявку | ✅ | ✅ | ✅ | ✅ |
| Смотреть свои заявки | ✅ | ✅ | ✅ | ✅ |
| Смотреть все заявки | ❌ | ✅ | ✅ | ✅ |
| Менять статус заявки | ❌ | ✅ | ✅ | ✅ |
| Отвечать в поддержку | ❌ | ✅ | ✅ | ✅ |
| Управлять товарами | ❌ | ❌/огр. | ✅ | ✅ |
| Управлять админами | ❌ | ❌ | ❌ | ✅ |
| Банить клиентов | ❌ | ❌/огр. | ✅ | ✅ |
| Настройки системы | ❌ | ❌ | ❌/огр. | ✅ |

Для MVP можно сделать проще: manager/admin/owner имеют доступ в админку, но важные действия логируются.

---

## 5. Правила доступа

1. Все admin callbacks должны проходить проверку роли.
2. Нельзя полагаться только на скрытые кнопки.
3. Если пользователь нажал старую admin-кнопку без прав — показать отказ.
4. Все изменения заявки через админку должны логироваться.
5. Заблокированный клиент не может создавать заявки и тикеты.

5. Добавить понятные сообщения отказа.

---

## 7. Что НЕ входит в MVP

Пока не делать:

```text
сложные permission-группы
настройку ролей из UI
историю всех входов
двухфакторную авторизацию
отдельный веб-кабинет сотрудников
```

---

## 8. Критерии готовности

Блок готов, если:

```text
обычный клиент не может попасть в админку
менеджер может работать с заявками и поддержкой
админ может управлять товарами
owner имеет полный доступ
запрещённые действия не выполняются даже через callback
важные действия логируются
```

---

## 9. Приоритет

Статус: база есть.

Приоритет: высокий для безопасности админки.

Нужно не усложнять роли, а закрыть очевидные дыры доступа.
