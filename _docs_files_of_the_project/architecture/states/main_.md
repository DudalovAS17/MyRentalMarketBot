# FSMStates

Общий справочник FSM-состояний проекта.

FSM в этом проекте используется только для краткоживущих сценариев Telegram-бота:
- создание товара в админке;
- создание заявки на аренду клиентом;
- поиск товаров;
- обращение в поддержку;
- ответ администратора на тикет;
- редактирование профиля;
- отдельные админские действия, где нужно дождаться текстового ввода.

FSM-состояния не являются доменной моделью, не хранятся в БД и не заменяют статусы из `status/*.py`.

## 0. Карта файлов `states/*.py`

| Файл | StatesGroup | Для чего |
|---|---|---|
| `states/admin.py` | `AdminStates` | Общие админские сценарии: поиск заявки/клиента, причины отклонения/отмены/бана. |

| `states/review.py` | `ReviewStates` | Черновой/будущий flow отзывов; активных handlers сейчас нет, код в `handlers/otzivi.py` закомментирован. |
| `states/search.py` | `SearchStates` | Пользовательский поиск товаров и пагинация результатов. |

| `states/user.py` | `ProfileEditStates` | Редактирование имени, email и телефона пользователя. |

## 1. Общие правила FSM

### Что хранится в FSM

FSM хранит временный контекст сценария:
- выбранные ID;
- черновики Pydantic-схем;
- номер страницы;
- ID сообщения, которое нужно редактировать;
- текстовые причины, которые пользователь/админ вводит в следующем сообщении.

### Что не должно храниться в FSM

- ORM-объекты SQLAlchemy.
- Долгоживущие бизнес-статусы.
- Права доступа.
- История действий.
- Данные, которые должны быть источником правды в БД.

### Когда делать `state.clear()`

`state.clear()` используется:
- при старте нового сценария, чтобы убрать хвосты старого flow;
- после успешного завершения сценария;
- при отмене пользователем;
- если из FSM невозможно достать обязательный контекст.

## 2. `AdminStates`

Файл: `states/admin.py`.

`AdminStates` — общий набор состояний для админских сценариев, которым нужно дождаться ввода текста/числа от сотрудника.

### Состояния

> `waiting_rental_id` — **ожидает ввод ID заявки**
> * Используется в админке заявок: «открыть заявку по ID».
> * Handler: `handlers/admin/deals.py`.
> * Ожидаемый ввод: текстовое число `rental_id`.
> * После успешного парсинга состояние очищается и показывается карточка заявки.

> `waiting_rental_cancel_reason` — **ожидает причину отмены подтверждённой заявки**
> * Используется в admin-flow отмены заявки компанией.
> * Handler: `handlers/admin/deals_status_actions.py`.
> * Перед входом в состояние в FSM кладётся `rental_id`.
> * Ожидаемый ввод: непустой текст причины.
> * Бизнес-переход: `CONFIRMED -> CANCELLED_BY_ADMIN`.

> `waiting_rental_reject_reason` — **ожидает причину отклонения заявки**
> * Используется в admin-flow отклонения заявки до подтверждения.
> * Handler: `handlers/admin/deals_status_actions.py`.
> * Перед входом в состояние в FSM кладётся `rental_id`.
> * Ожидаемый ввод: непустой текст причины.
> * Бизнес-переход: `REQUESTED/IN_PROGRESS -> REJECTED`.

> `waiting_item_reject_reason` — **ожидает причину отклонения/модерационного решения по товару**
> * Состояние определено в `states/admin.py`.
> * В актуальных handlers прямого активного использования почти нет: item-flow сейчас работает через `ItemStatus` (`DRAFT`, `ACTIVE`, `HIDDEN`, `ARCHIVED`) и кнопки админки.
> * Оставлено как задел под сценарий, где админ должен написать причину по товару.

> `waiting_user_id` — **ожидает ввод user_id для поиска клиента**
> * Используется в админке пользователей.
> * Handler: `handlers/admin/users_moderation.py`.
> * Ожидаемый ввод: текстовое число `user_id`.
> * После успешного парсинга состояние очищается и показывается карточка пользователя.

> `waiting_user_ban_reason` — **ожидает причину бана клиента**
> * Используется при бане клиента администратором.
> * Handler: `handlers/admin/users_moderation.py`.
> * Перед входом в состояние в FSM кладётся `target_user_id`.
> * Ожидаемый ввод: непустой текст причины.
> * После применения бана состояние очищается.

### Временные ключи FSM

| Ключ | Где используется | Что хранит |
|---|---|---|
| `rental_id` | reject/cancel rental flow | ID заявки, для которой ждём причину. |
| `target_user_id` | ban user flow | ID клиента, которого нужно забанить. |

## 3. `AdminSupportStates`

Файл: `states/admin_support.py`.

### Состояния

> `waiting_reply_text` — **ожидает текст ответа администратора на тикет**
> * Используется в админской поддержке.
> * Handler: `handlers/admin/support.py`.
> * Перед входом в состояние в FSM кладётся `ticket_id`.

> * После успешной доставки ответа пользователю и фиксации `admin_last_reply_at` состояние очищается.

### Временные ключи FSM

| Ключ | Что хранит |
|---|---|
| `ticket_id` | ID открытого тикета поддержки, по которому админ пишет ответ. |

### Особенность

Ответ администратора не закрывает тикет автоматически. Закрытие тикета — отдельное callback-действие в админке поддержки.

## 4. `ItemCreateStates`

Файл: `states/item.py`.

`ItemCreateStates` описывает создание товара каталога компании в админке.

Это не создание пользовательского объявления. Товар создаёт сотрудник компании, затем товар попадает в каталог со статусом по бизнес-логике сервиса.

### Состояния

> `category` — **ожидает выбор категории**
> * Стартовый шаг после кнопки «создать товар».
> * Handler: `handlers/admin/create_item.py`.
> * Ожидаемый ввод: callback категории.
> * FSM инициализируется ключами `mode`, `admin_user_id`, `new_item`, `photos`.

> `subcategory` — **ожидает выбор подкатегории**
> * Используется после выбора категории.
> * Ожидаемый ввод: callback подкатегории.
> * В FSM сохраняются выбранные category/subcategory и обновляется `new_item`.

> `title` — **ожидает название товара**
> * Ожидаемый ввод: текст.
> * Валидируется helper-ами create item flow.
> * При ошибке FSM остаётся в `title`.

> `description` — **ожидает описание товара**
> * Ожидаемый ввод: текст.
> * При успехе описание пишется в `new_item`, затем переход к цене.

> `price` — **ожидает цену аренды**
> * Ожидаемый ввод: денежное значение текстом.
> * При успехе цена пишется в `new_item`, затем переход к количеству.

> `available_quantity` — **ожидает количество товара в наличии**
> * Актуальная замена старого шага `deposit`.
> * Ожидаемый ввод: целое количество.
> * При успехе пишется в `new_item.available_quantity`.

> `rental_period` — **ожидает минимальный срок аренды**
> * Ожидаемый ввод: число дней/текст, который парсится helper-ом.
> * При успехе пишется в `new_item.min_rental_period`.

> `photos` — **ожидает фотографии товара**
> * Ожидаемый ввод:
>   - фото;
>   - либо текстовая кнопка `✅ Готово`.
> * Фото сохраняются отдельно от `new_item` в ключ `photos` как список Telegram `file_id`.
> * Есть ограничение `ADMIN_MAX_PHOTOS`.

> `confirmation` — **ожидает подтверждение публикации/создания товара**
> * Показывается итоговый preview товара.
> * Ожидаемый ввод: callback публикации или отмены.
> * При подтверждении создаётся `Item`, затем отдельно прикрепляются фото.
> * После успеха или отмены FSM очищается.

### Временные ключи FSM

| Ключ | Что хранит |
|---|---|
| `mode` | Режим сценария, сейчас используется `ADMIN_CREATE_ITEM_MODE`. |
| `admin_user_id` | Внутренний ID пользователя/админа из middleware `user`. |
| `new_item` | Черновик `ItemCreateDraft` в виде dict. |
| `photos` | Список Telegram `file_id` фотографий. |
| `selected_category_id` | Выбранная категория. |
| `selected_category_name` | Название выбранной категории для UI. |
| `selected_subcategory_id` | Выбранная подкатегория. |
| `selected_subcategory_name` | Название выбранной подкатегории для UI. |

### Важное отличие от старой документации

Старые поля `deposit` и `location` в актуальном `ItemCreateStates` отсутствуют. Вместо залога есть `available_quantity`, а location сейчас не часть FSM создания товара.

## 5. `RentalCreateStates`

Файл: `states/rental.py`.

`RentalCreateStates` описывает создание клиентской заявки на аренду товара.

### Состояния

> `period` — **ожидает выбор фиксированного периода аренды**
> * Стартует после callback `RENT_ITEM_CB` из карточки товара.
> * Handler: `handlers/rentals/flow_create.py`.
> * Ожидаемый ввод: callback `RENT_PERIOD_CB...`.
> * В FSM уже лежит `rent_draft` с `item_id`, `client_name`, `client_phone`.

> `comment` — **ожидает детали заявки текстом**
> * Используется после выбора периода.
> * Ожидаемый ввод: сообщение в формате, который парсит `parse_rent_details_message(...)`.
> * Сейчас ожидается формат вида:
>   `3 - 25.06.2026 - Нужна доставка вечером`.
> * В draft сохраняется клиентский комментарий и итоговый текст периода.

> `confirmation` — **ожидает подтверждение заявки**
> * Показывает итоговую информацию по заявке.
> * Ожидаемый ввод: callback подтверждения `CONFIRM_RENT_CB` или отмены flow.
> * При подтверждении создаётся `RentalCreate`, затем `RentalService.create(...)`.
> * После успешного создания отправляются уведомления клиенту и админам, затем FSM очищается.

### Временные ключи FSM

| Ключ | Что хранит |
|---|---|
| `rent_draft` | Черновик `RentalCreateDraft` в JSON/dict виде. |
| `rent_ui_message_id` | ID сообщения-экрана аренды, которое flow редактирует через `render_rent_ui(...)`. |

### Отмена

Callback `CANCEL_RENT_FLOW_CB`:
- берёт `rent_ui_message_id` и `item_id` из FSM;
- перерисовывает экран отмены;
- очищает FSM.

### Важное отличие от старой документации

Старых состояний `start_date` и `end_date` сейчас нет. Актуальный flow использует:
- `period` — выбор фиксированного периода;
- `comment` — ввод деталей/даты/комментария одним сообщением;
- `confirmation` — финальное подтверждение.

## 6. `SearchStates`

Файл: `states/search.py`.

### Состояния

> `waiting_for_query` — **ожидает поисковый запрос пользователя**
> * Стартует через `/search`, кнопку `🔎 Поиск` или callback нового поиска.
> * Handler: `handlers/search/search.py`.
> * Ожидаемый ввод: текст запроса.
> * При невалидном запросе FSM остаётся в этом состоянии.

> `browsing` — **пользователь просматривает результаты поиска**
> * Устанавливается после валидного запроса.
> * Используется для пагинации и возврата к результатам.
> * Если query потерян, flow возвращает пользователя к `waiting_for_query`.

### Временные ключи FSM

| Ключ | Что хранит |
|---|---|
| `search_query` | Нормализованный текст поиска. |
| `search_page` | Текущая страница результатов. |

## 7. `SupportStates`

Файл: `states/support_ticket.py`.

### Состояния

> `waiting_text` — **ожидает текст обращения в поддержку**
> * Стартует через `/support`, callback `support:start` или из карточки заявки.
> * Handler: `handlers/support/support.py`.
> * Перед входом проверяется, что у пользователя нет открытого тикета.
> * Ожидаемый ввод: непустой текст обращения.
> * После создания тикета FSM очищается, клиент и админы получают уведомления.

### Временные ключи FSM

| Ключ | Что хранит |
|---|---|
| `support_rental_id` | ID заявки, если обращение начато из карточки заявки; иначе `None`. |

### Особенности

- Если у пользователя уже есть открытый тикет, FSM не запускается.
- Если тикет успели создать параллельно, ловится `TicketAlreadyOpen`, FSM очищается и пользователю показывается уже открытый тикет.
- Отмена через callback `support:cancel` очищает FSM и возвращает пользователя в главное меню.

## 8. `ProfileEditStates`

Файл: `states/user.py`.

### Состояния

> `waiting_for_name` — **ожидает новое имя пользователя**
> * Стартует из настроек профиля.
> * Handler: `handlers/auth/edit_profile.py`.
> * Ожидаемый ввод: текст имени.
> * При успехе обновляется `User.full_name`, FSM очищается.

> `waiting_for_email` — **ожидает новый email пользователя**
> * Ожидаемый ввод: текст email.
> * При успехе обновляется `User.email`, FSM очищается.

> `waiting_for_phone` — **ожидает новый телефон пользователя**
> * Ожидаемый ввод: Telegram contact через кнопку `request_contact=True`.
> * Handler проверяет, что контакт принадлежит самому пользователю.
> * Неверный ввод не очищает FSM, а повторно просит отправить контакт через кнопку.

### Особенность

Состояния профиля заменяют старые флаги вида `editing_field_profile` / `changing_phone`. Контекст редактирования хранится в FSM, а итоговые данные сохраняются через `UserService.update(...)`.

## 9. `ReviewStates`

Файл: `states/review.py`.

### Состояния

> `rating` — **ожидает оценку отзыва**
> * Предполагаемый ввод: callback рейтинга.

> `comment` — **ожидает комментарий к отзыву**
> * Предполагаемый ввод: текст комментария или callback пропуска.

### Текущий статус реализации

`ReviewStates` определён, но активный flow отзывов сейчас не подключён: код в `handlers/otzivi.py` находится внутри многострочного комментария и относится к старой owner/renter-логике.

Актуальная доменная модель отзывов уже есть в schemas/services/repositories, но FSM-flow для клиентского создания отзыва нужно переписать под текущую модель:
- отзыв по завершённой заявке компании;
- `rental_id`, `user_id`, опционально `item_id`;
- `ReviewStatus.PENDING` после создания;
- без `reviewee_id/owner_id/renter_id` из старого marketplace-сценария.

## 10. Дополнительные FSM-данные вне `states/*.py`

Некоторые handlers используют `FSMContext` как короткое хранилище даже без отдельного `StatesGroup`.

### Каталог

Файлы `handlers/category/...` сохраняют:
- выбранную категорию;
- выбранную подкатегорию;
- выбранный товар.

Типичные ключи:
- `selected_category_id`;
- `selected_category_name`;
- `selected_subcategory_id`;
- `selected_subcategory_name`;
- `selected_item_id`.

### Админский список товаров

Админские helpers могут сохранять:
- `admin_items_page`;
- `admin_items_status`.

Это не отдельный wizard, а UI-контекст пагинации/фильтра.

## 11. Общая архитектурная оценка

### Что хорошо

- FSM разделён по сценариям: admin, admin support, item create, rental create, search, support, profile edit.
- FSM не смешан с ORM/repository.
- Большинство состояний короткие и завершаются через `state.clear()`.
- Бизнес-статусы (`ItemStatus`, `RentalStatus`, `ReviewStatus`) вынесены отдельно в `status/*.py`, а FSM хранит только шаги диалога.

### Что важно контролировать

- Не держать в FSM устаревшие marketplace-понятия (`owner_id`, `renter_id`, `reviewee_id`) в новых flow.
- При добавлении нового шага обновлять и `states/*.py`, и handlers, и этот документ.
- При старте нового большого flow всегда очищать старый FSM-контекст.
- Не хранить ORM-объекты в FSM; хранить ID или Pydantic-draft в dict/json виде.
- Не забывать очищать состояние при ошибке, если контекст стал непригоден.
