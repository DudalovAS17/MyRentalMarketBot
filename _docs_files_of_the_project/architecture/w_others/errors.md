# Error handling и логирование

Документ фиксирует актуальный контракт обработки ошибок в проекте.

Главная идея: ожидаемые бизнес-отказы превращаются в понятный UX в handler-ах, а неожиданные технические ошибки уходят в единый глобальный middleware.

---

## 1) Классы ошибок

Бизнес-ошибки находятся в `utils/errors.py`:

- `ServiceError` — базовый тип ожидаемой ошибки business-layer;
- `NotFoundError` — сущность не найдена в строгом сценарии;
- `ForbiddenError` — действие запрещено правилами продукта;
- `ConflictError` — конфликт текущего состояния, статуса или инварианта;
- `ValidationError` — бизнес-валидация не пройдена.

Доменные сигналы с полезными данными находятся в `utils/domain_exceptions.py`, например:

- `ItemNotAvailable` — товар заблокирован активной арендой;
- `TicketAlreadyOpen` — у пользователя уже есть открытый тикет поддержки.

---

## 2) Handler

Handler отвечает за UX, поэтому он:

1. Быстро отвечает на callback (`callback.answer()`), чтобы убрать “часики”.
2. Парсит входные данные (`callback.data`, текст, FSM draft).
3. Ловит только ожидаемые бизнес-ошибки (`ServiceError` и наследники / доменные сигналы), если нужно показать человеку конкретный текст.
4. Не ловит `Exception` как штатный путь: технические ошибки перехватывает `GlobalErrorMiddleware`.

Некорректный `callback_data` — это ошибка входа/старой кнопки, она обрабатывается прямо в handler-е:

```python
try:
    item_id = int(callback.data.split(":")[-1])
except (AttributeError, IndexError, ValueError):
    await callback.answer("Некорректная кнопка. Откройте экран заново.", show_alert=True)
    return
```

Ожидаемый бизнес-отказ:

```python
from utils.errors import NotFoundError, ServiceError

try:
    item = await item_service.get_item_by_id(item_id, strict=True)
except NotFoundError:
    await callback.answer("Товар не найден", show_alert=True)
    return
except ServiceError as exc:
    await callback.answer(str(exc), show_alert=True)
    return
```

Если действие меняет состояние, после успеха нужно перерисовать экран/детали. После бизнес-отказа перерисовка тоже желательна, если пользователь нажал устаревшую inline-кнопку и экран можно безопасно обновить.

---

## 3) Service

Service — источник бизнес-решений.

Правила:

- service поднимает только ожидаемые бизнес-ошибки (`utils.errors`) или доменные сигналы (`utils.domain_exceptions`);
- технические ошибки БД/сети/зависимостей не подавляются;
- `strict=False` возвращает `None` / `False` для допустимого отсутствия результата; (см. 8 ниже)
- `strict=True` превращает отсутствие обязательной сущности или конфликт в `ServiceError`-наследника. (см. 8 ниже)

Логирование в service нужно для бизнес-событий и аудита.
### [Что именно логировать в сервисах]()
    1) Переходы статусов (товары / заявки / пользователи)
        кто инициатор (admin_id / user_id)
        что изменилось (old_status → new_status)
        сущность (item_id / rental_id / target_user_id)
        причина (если есть)

    2) Инварианты / Security / Guards
        попытка запретного действия (например скрыть item при open rentals)
        попытка бана админа/самобана
        попытка действия от BANNED (но у тебя это в middleware — там лог ок)

    3) Админские actions
        Админка — это всегда аудитопригодная зона. Там лог нужен почти всегда.

Не нужно логировать каждую техническую CRUD-операцию без бизнес-смысла.

Примеры:
```
- ❌ Не логируем “Example создан id=...”, если это просто техническая сущность без бизнес-события.
- ✅ логируем, если создание = бизнес-факт.

- RentalService.create_rental() → это бизнес-событие “создана сделка” → лог MUST
- PhotoRepository.create() → это инфраструктура → лог не нужен
```

## 4) Repository

Repository — слой БД, он не решает UX и бизнес-политику.

Правила:

- не ловить `Exception` “на всякий случай”;
- не логировать успешные read/write операции как бизнес-события;
- не возвращать DTO;
- нормализовать только конкретные DB-ошибки, если это явно часть контракта метода;
- write-операции используют commit helpers из `BaseRepository`.

В актуальном коде rollback централизован в `BaseRepository._commit_or_rollback()`: commit выполняется в `try`, при ошибке делается rollback, затем исключение пробрасывается выше.

---

## 5) GlobalErrorMiddleware

`middlewares/global_error.py` — единая страховка для неожиданных технических ошибок.

Он:

- пропускает `ServiceError` дальше, чтобы handler мог обработать ожидаемый бизнес-отказ;
- логирует traceback для неожиданных `Exception` через `logger.exception(...)`;
- показывает пользователю нейтральный текст из `texts/text_middleware.py`;
- не раскрывает технические детали пользователю.

Это позволяет не засорять каждый handler одинаковыми `except Exception` блоками.

---

## 6) Telegram render errors

Ошибки редактирования сообщения (`TelegramBadRequest`) относятся к UI/render layer.

Рекомендуемая политика для render/helper-функций:

- `message is not modified` — можно игнорировать;
- `message can't be edited`, `message to edit not found`, `there is no text in the message to edit` — отправить новое сообщение, если UX это допускает;
- не превращать каждую ошибку рендера в бизнес-ошибку service-layer.

---

## 7) Короткий чеклист

- [ ] Handler ловит только ожидаемые business/domain errors и ошибки парсинга входа.
- [ ] Handler не ловит broad `Exception` без крайней необходимости.
- [ ] Service принимает бизнес-решения и поднимает `ServiceError`-наследников.
- [ ] Repository не содержит user-facing текстов и не подавляет DB-исключения.
- [ ] Технические ошибки централизованно логирует `GlobalErrorMiddleware`.
- [ ] После успешных callback-action экран обновляется, если inline-кнопки должны измениться.

---

## 8) **[Service Strict-mode rule]()**
Это правило поведения сервиса, когда сущность не найдена в репозитории.

Репозиторий — тупой слой БД. Он может вернуть: None | False. 
Но он не знает, это допустимая ли это ситуация или ошибка бизнес-сценария?
👉 Решать обязан сервис.

>Сценарий 1: *strict=False (мягкий режим)*: вернуть None, если не найдено.

Если категории нет → возвращаем `None`. Это допустимое отсутствие результата, например:
- список
- фильтр
- необязательный объект

>Сценарий 2: *strict=True (жёсткий режим)*: если не найдено -> utils.errors (ошибка бизнес-сценария).

Если категории нет → бросаем: `raise NotFoundError(...)`. Потому что:
- в этом сценарии категория обязана существовать
- её отсутствие — бизнес-ошибка
- отсутствие результата считается ошибкой бизнес-сценария и приводит к исключению


*Технические исключения (DB errors) не перехватываются strict и пробрасываются выше.*
