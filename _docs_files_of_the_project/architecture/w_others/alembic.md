## Миграции (Alembic)

### **Создать миграцию:**
alembic revision --autogenerate -m "message"

Появится миграция.
- SQL на создание всех таблиц
- enum’ы
- индексы
- FK

### **Применить миграции:**
alembic upgrade head

После этого в БД появятся:
- все таблицы
- таблица alembic_version

### **Откатить на одну:**
alembic downgrade -1

# 📁 alembic/

### 📄 alembic/env.py — САМОЕ ВАЖНОЕ

- связывает Alembic с твоим проектом
- говорит:
    - откуда брать DATABASE_URL
    - где Base.metadata
    - какие модели учитывать при автогенерации
- управляет online / offline режимом миграций

👉 90% проблем с Alembic — это env.py
Если он корректен — всё остальное работает.

Тут будут ВСЕ миграции.

❗ Никогда не удаляй файлы из versions/ вручную, если миграции уже применялись.

🔹 **import db.models**

Без этого Alembic не увидит таблицы.
Это обязательное правило.

🔹 **target_metadata = Base.metadata**

Это единственный источник правды о схеме БД.

🔹 **compare_type=True**

Alembic будет:
- отслеживать изменения типов
- enum’ов
- Numeric → Decimal
- Boolean / nullable
- Без этого миграции будут неполными

## import db.models

ВАЖНО: импортируем ВСЕ модели, чтобы Alembic увидел их в metadata
Если модели импортируются лениво — обязательно импортни пакет моделей,
чтобы Base увидел все таблицы. Например:

import db.models  # noqa: F401

Если твои модели разбросаны по файлам и не импортируются автоматически,
Alembic “не увидит” таблицы.
Поэтому в env.py нужно импортнуть пакет моделей (или конкретные модули),
чтобы Base.metadata наполнилась.
