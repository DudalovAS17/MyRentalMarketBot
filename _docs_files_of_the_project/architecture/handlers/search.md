# Search — поиск товаров

Раздел описывает текущий поиск по каталогу товаров компании. Поиск не создаёт заявки и не меняет данные; он только помогает пользователю найти активные товары и перейти к карточке товара.

---

## 1. Карта файлов

| Файл                                | Назначение                                                                                      |
|-------------------------------------|-------------------------------------------------------------------------------------------------|
| `handlers/search/search.py`         | `search_router`, FSM поиска, пагинация результатов.                                             |
| `handlers/search/helpers_search.py` | Нормализация, валидация запроса, построение текстов и клавиатур, загрузка страницы результатов. |

---

## 2. `start_search`: Сценарий поиска

1. Пользователь отправляет `/search` или нажимает `🔎 Поиск`.
2. `start_search()` вызывает `prompt_search_query()`.
3. FSM переходит в `SearchStates.waiting_for_query`.
4. Пользователь вводит текст запроса.
5. `process_search_query()` нормализует и валидирует запрос.
6. FSM переходит в `SearchStates.browsing`.
7. `show_search_results()` показывает первую страницу.
8. Callback-и пагинации открывают следующие/предыдущие страницы.

---

## 3. `request_new_query`: Запрос нового поиска

`request_new_query()` срабатывает по callback `SEARCH_NEW_QUERY_CB`.

Он сбрасывает текущий поисковый контекст:

- `search_query=None`;
- `search_page=1`;
- состояние `SearchStates.waiting_for_query`.

---

## 4. `process_search_query()`: Ввод поискового запроса

`process_search_query()` принимает текст только в состоянии `waiting_for_query`.

Обработка:

- `normalize_search_query()` обрезает и нормализует ввод;
- `validate_search_query()` проверяет минимальные требования;
- при ошибке FSM остаётся в том же состоянии;
- при успехе в FSM сохраняются `search_query` и `search_page=1`;
- вызывается `show_search_results(..., page=1)`.

---

## 5. `paginate_search()`: Пагинация

`paginate_search()` обрабатывает callback-и с `SEARCH_PAGE_CB_PREFIX`.

Он:

1. парсит страницу через `parse_search_page()`;
2. проверяет, что в FSM есть `search_query`;
3. если запроса нет — возвращает пользователя к prompt;
4. вызывает `show_search_results()` для нужной страницы.

`back_to_search_results()` по `SEARCH_BACK_CB` просто перерисовывает текущую страницу результатов из FSM.

---

## 6. `show_search_results()`: Показ результатов

`show_search_results()`:

- берёт `search_query` и текущую страницу из FSM;
- вызывает `fetch_search_page(query, current_page, item_service)`;
- при ошибке сервиса показывает безопасную ошибку;
- если товаров нет — показывает пустое состояние;
- если товары есть — показывает список и клавиатуру `build_search_keyboard()`;
- обновляет `search_page` на безопасную страницу, возвращённую helper-ом.

Результаты поиска должны вести к карточке товара через callback-и каталога/деталей, а не создавать отдельный параллельный item flow.

## 8. Helpers

`helpers_search.py` отвечает за:

- нормализацию запроса;
- валидацию;
- парсинг страницы;
- тексты prompt/empty/results;
- клавиатуры;
- `fetch_search_page()` — единый вызов `ItemService` для страницы результатов.

Это позволяет держать handler коротким и не дублировать форматирование.

---

## 9. Правила разработки

1. Поиск должен работать только с публичными/доступными товарами — фильтр держать в `ItemService`.
2. Не хранить найденные товары в FSM; хранить только query/page.
3. Все callback-и пагинации должны быть устойчивы к отсутствующему FSM-контексту.
4. При добавлении фильтров поиска хранить в FSM только параметры фильтра, а не ORM/DTO-объекты.
5. Переход к товару должен использовать общий item/category UI, чтобы не плодить разные карточки одного товара.
