# FSMStates

[**Общий справочник FSM-состояний проекта**]()

Файлы состояний описывают временные сценарные шаги бота: 
- создание объявлений
- оформление аренды
- поиск
- поддержку
- редактирование профиля
- отзывы
- admin-flow 

Все состояния наследуются от `StatesGroup` и используются как краткоживущий FSM-контекст, 
а не как доменная модель.

---

# AdminStates

[**FSM-состояния общих admin-сценариев**]()

### Состояния:

> `waiting_rental_id` - **[Ожидает ввод ID сделки админом]()**
> * Используется для сценария открытия сделки по ID.
> * Ожидаемый ввод: `rental_id`

> `waiting_cancel_reason` - **[Ожидает причину отмены сделки]()**
> * Используется в admin-flow отмены сделки.
> * Ожидаемый ввод: текст причины отмены

> `waiting_dispute_resolution` - **[Ожидает текст решения по спору]()**
> * Используется при закрытии спорной сделки.
> * Ожидаемый ввод: краткое решение администратора

> `waiting_dispute_target` - **[Ожидает выбор итогового статуса спора]()**
> * Используется после ввода решения по спору.
> * Ожидаемый ввод: callback выбора итогового статуса

> `waiting_item_reject_reason` - **[Ожидает причину отклонения объявления]()**
> * Используется в admin-flow модерации объявлений.
> * Ожидаемый ввод: текст причины отклонения

> `waiting_user_id` - **[Ожидает ввод user_id для поиска пользователя]()**
> * Используется в admin-flow управления пользователями.
> * Ожидаемый ввод: текстовое число `user_id`

> `waiting_user_ban_reason` - **[Ожидает причину бана пользователя]()**
> * Используется при бане пользователя администратором.
> * Ожидаемый ввод: текст причины бана

* Для MVP нормально, но при росте можно разделить на:
  - `AdminDealStates`
  - `AdminItemStates`
  - `AdminUserStates`

# AdminSupportStates

[**FSM-состояния admin-ответов на тикеты поддержки**]()

### Состояния:

> `waiting_reply_text` - **[Ожидает текст ответа администратора на тикет]()**
> * Используется в admin-support flow.
> * Ожидаемый ввод: текст ответа пользователю

* Состояние выделено отдельно от обычных `AdminStates`.
* Это хорошо, потому что admin-support — отдельный сценарий.

# ItemCreateStates

[**FSM-состояния создания объявления**]()

### Состояния:

> `category` - **[Ожидает выбор категории объявления]()**
> * Первый шаг создания объявления.
> * Ожидаемый ввод: callback выбора категории

> `subcategory` - **[Ожидает выбор подкатегории]()**
> * Используется после выбора категории.
> * Ожидаемый ввод: callback выбора подкатегории

> `title` - **[Ожидает название объявления]()**
> * Ожидаемый ввод: текст названия

> `description` - **[Ожидает описание объявления]()**
> * Ожидаемый ввод: текст описания

> `price` - **[Ожидает цену аренды]()**
> * Ожидаемый ввод: числовое значение цены

> `deposit` - **[Ожидает размер залога]()**
> * Ожидаемый ввод:
>   - числовое значение залога
>   - либо пропуск, если сценарий поддерживает отсутствие залога

> `location` - **[Ожидает местоположение объявления]()**
> * Ожидаемый ввод: текст города / района / локации

> `rental_period` - **[Ожидает период аренды]()**
> * Ожидаемый ввод: текстовый или callback-выбор периода аренды

> `photos` - **[Ожидает фотографии объявления]()**
> * Ожидаемый ввод:
>   - фото
>   - команда/кнопка завершения загрузки фото

> `confirmation` - **[Ожидает подтверждение создания объявления]()**
> * Финальный шаг перед созданием объявления.
> * Ожидаемый ввод:
>   - callback подтверждения / отмены

# RentStates

[**FSM-состояния оформления аренды**]()

### Состояния:

> `start_date` - **[Ожидает выбор даты начала аренды]()**
> * Пользователь выбирает дату начала аренды.

> `end_date` - **[Ожидает выбор даты окончания аренды]()**
> * Пользователь выбирает дату окончания аренды.

> `confirmation` - **[Ожидает подтверждение заявки на аренду]()**
> * Финальный шаг перед отправкой запроса аренды.

# ProfileEditStates

[**FSM-состояния редактирования профиля пользователя**]()

### Состояния:

> `waiting_for_name` - **[Ожидает новое имя пользователя]()**
> * Используется при изменении имени в профиле.
> * Ожидаемый ввод: текст имени

> `waiting_for_email` - **[Ожидает новый email пользователя]()**
> * Используется при изменении email.
> * Ожидаемый ввод: текст email

> `waiting_for_phone` - **[Ожидает новый контакт-телефон пользователя]()**
> * Используется при смене номера телефона.
> * Ожидаемый ввод: Telegram contact через `request_contact=True`

* Состояния правильно заменяют старые флаги вроде `editing_field_profile` и `changing_phone`.

# Общая архитектурная оценка

### Что хорошо:

- FSM-состояния разделены по крупным сценариям:
  - admin;
  - admin support;
  - item create;
  - rent create;
  - review;
  - search;
  - user support;
  - profile edit.
- Нет смешивания FSM с БД / ORM / repository.
- Состояния не содержат бизнес-логики.
- Большинство `StatesGroup` имеют понятные имена и отражают реальные пользовательские flow.
