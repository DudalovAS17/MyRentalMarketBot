# RentalMarketBot — стратегическая основа MVP

> **Рабочее определение проекта:**  
> RentalMarketBot — это не обычный Telegram-бот и не C2C-маркетплейс.  
> Это **Telegram-система заявок для компании по аренде оборудования**: каталог товаров, карточки, заявки клиентов, менеджерская админка, поддержка и уведомления.

---

## 1. Главная идея проекта

RentalMarketBot должен решать конкретную бизнес-задачу:

```text
Клиенту удобно выбрать оборудование
→ быстро оставить заявку
→ менеджер получает структурированную заявку
→ компания не теряет обращения из Telegram
```

Проект нужно мыслить не как набор handlers, а как небольшую бизнес-систему внутри Telegram:

```text
Telegram-витрина компании
+ каталог аренды
+ система сбора заявок
+ мини-CRM для менеджера
+ поддержка клиентов
+ база товаров
+ уведомления
```

Главная ценность бота — не в том, что он просто показывает товары, а в том, что он превращает хаотичные сообщения клиентов в понятные заявки для менеджера.

---

## 2. Продуктовая формула

Финальная формула проекта:

```text
RentalMarketBot =
каталог аренды оборудования
+ карточки товаров
+ заявки клиентов
+ админка менеджера
+ поддержка
+ уведомления
```

Или ещё короче:

```text
Каталог → Заявка → Менеджер → Статус → Ответ клиенту
```

---

## 3. Что НЕ является целью MVP

На первом этапе не нужно строить большой маркетплейс, интернет-магазин или полноценную CRM.

Для MVP не нужны:

- онлайн-оплата;
- корзина на несколько товаров;
- избранное;
- отзывы;
- сложная аналитика;
- веб-админка;
- Telegram Mini App;
- автоматический расчёт сложной стоимости;
- полноценный складской учёт;
- интеграция с 1С/CRM;
- рекомендации и персонализация.

MVP должен доказать главный сценарий:

```text
Пользователь нашёл товар
→ оставил заявку
→ менеджер получил заявку
→ менеджер обработал заявку
→ пользователь получил ответ
```

---

## 4. Основные контуры системы

Стратегически бот состоит из пяти крупных контуров:

```text
1. Клиентский каталог
2. Заявка на аренду
3. Менеджерская обработка
4. Поддержка и коммуникация
5. Аналитика / развитие / коммерческая упаковка
```

---

# 5. Контур 1 — клиентская часть

## Назначение

Клиентская часть — это всё, что видит обычный пользователь.

Пользователь не должен разбираться в структуре бота. Его путь должен быть коротким, понятным и предсказуемым.

## Основные элементы

```text
/start
главное меню
каталог аренды
категории
подкатегории
карточки товаров
мои заявки
условия аренды
доставка и оплата
поддержка
связаться с менеджером
```

## Главный пользовательский путь

```text
/start
→ Главное меню
→ Каталог аренды
→ Категория
→ Подкатегория
→ Карточка товара
→ Оставить заявку
→ Количество
→ Срок аренды
→ Доставка
→ Контакты
→ Комментарий
→ Заявка отправлена менеджеру
```

## Принцип UX

Клиентская часть должна быть простой:

```text
минимум текста
понятные кнопки
короткие шаги
возможность вернуться назад
ясное подтверждение действия
```

# 6. Контур 2 — каталог товаров

## Назначение

Каталог — это ядро проекта.  
Если каталог слабый, бот не будет выглядеть как продукт.

## Структура каталога

```text
Category
→ Subcategory
→ Item
→ ItemPhoto
→ ItemCharacteristic
```

## Карточка товара

Карточка товара должна работать как мини-страница товара:

```text
фото
название
цена
описание
характеристики
наличие
статус
кнопка "Оставить заявку"
кнопка "Задать вопрос"
переключение фото
переключение между товарами
```

## Важное UX-решение

Товары лучше показывать не длинным списком, а каруселью:

```text
← Предыдущий товар | Оставить заявку | Следующий товар →
```

Это особенно важно для Telegram, потому что длинные списки быстро превращаются в мусорный поток сообщений.

## MVP-решение

Для MVP можно не делать полноценное добавление товаров через админку.  
Допустимо начать с seed-данных:

```text
заранее созданные категории
заранее созданные товары
заранее добавленные фото
заранее заполненные характеристики
```

Это позволит быстрее собрать рабочий демо-сценарий.

# 7. Контур 3 — заявка на аренду

## Назначение

Заявка — это главное бизнес-событие в системе.

В проекте `Rental` нужно воспринимать не как уже начавшуюся аренду, а как **заявку клиента на аренду**.

## Почему не оплата в MVP

Для аренды строительного оборудования реалистичнее ручное подтверждение менеджером:

```text
клиент оставил заявку
→ менеджер проверил наличие
→ уточнил дату, срок, доставку, залог
→ подтвердил или отклонил заявку
```

Онлайн-оплата может появиться позже, но для MVP она не является обязательной.

## Форма заявки

Минимальная форма:

```text
товар
количество
срок аренды
нужна ли доставка
адрес доставки, если нужна
имя клиента
телефон клиента
комментарий
```

## FSM заявки

Пример сценария:

```text
Выбран товар
↓
Количество
↓
Срок аренды
↓
Доставка: да/нет
↓
Адрес, если доставка нужна
↓
Имя
↓
Телефон
↓
Комментарий
↓
Подтверждение заявки
↓
Создание Rental со статусом REQUESTED
↓
Уведомление менеджеру
```

## Результат

После создания заявки:

```text
клиент получает подтверждение
менеджер получает уведомление
заявка появляется в админке
```

# 8. Контур 4 — менеджерская админка

## Назначение

Админка превращает бота из простой витрины в рабочий инструмент компании.

Без админки бот — это каталог.  
С админкой бот — это система обработки заявок.

## Роли

Минимально:

```text
owner
admin
manager
```

В MVP можно начать проще:

```text
admin / manager
```

## Основные разделы админки

```text
новые заявки
все заявки
детали заявки
изменение статуса
комментарии менеджера
товары
категории
поддержка
клиенты
статистика
настройки
```

## MVP админки

Для первой версии важнее всего не управление товарами, а обработка заявок:

```text
🆕 Новые заявки
📋 Все заявки
🔎 Детали заявки
✅ Взять в работу
✅ Подтвердить
❌ Отклонить
🏁 Завершить
📞 Связаться с клиентом
📝 Добавить комментарий
```

## Статусы заявки

MVP-статусы:

```text
REQUESTED              — заявка создана клиентом
IN_PROGRESS            — менеджер взял заявку в работу
CONFIRMED              — заявка подтверждена
REJECTED               — заявка отклонена
COMPLETED              — заявка завершена
CANCELLED_BY_CLIENT    — отменена клиентом
CANCELLED_BY_ADMIN     — отменена админом
```

## Главная цель админки

Менеджер должен видеть не просто сообщение, а структурированную карточку заявки:

```text
номер заявки
клиент
телефон
товар
количество
срок
доставка
адрес
комментарий клиента
статус
дата создания
кнопки действий
```

# 9. Контур 5 — поддержка

## Назначение

Поддержка нужна, потому что в аренде клиент часто задаёт уточняющие вопросы:

```text
подойдёт ли товар?
есть ли доставка?
есть ли в наличии?
можно ли забрать сегодня?
какой залог?
что входит в комплект?
можно ли арендовать на сутки?
```

## Минимальная модель поддержки

```text
SupportTicket
SupportMessage
```

Тикет может быть связан с:

```text
пользователем
товаром
заявкой
```

## MVP-сценарии поддержки

```text
Пользователь задаёт вопрос
→ создаётся тикет
→ менеджер получает уведомление
→ менеджер отвечает
→ пользователь получает ответ
→ тикет закрывается
```

## Важно

Поддержка не должна быть просто свободным чатом.  
Лучше, если вопрос можно привязать к товару или заявке.

# 10. Архитектура по слоям

## Главный принцип

Handlers не должны содержать всю бизнес-логику.  
Они должны только принимать действие пользователя и вызывать нужный service.

Правильный поток:

```text
message/callback
↓
handler
↓
service
↓
repository
↓
database
```

А затем обратно:

```text
database
↓
repository
↓
service
↓
handler
↓
text/keyboards/notification
```

# 13. Поток создания заявки

Правильный поток:

```text
Пользователь нажал "Оставить заявку"
↓
handler получил callback
↓
CatalogService проверил товар
↓
FSM начал сбор данных
↓
handler собрал количество, срок, доставку, контакты
↓
RentalService создал заявку
↓
RentalRepository сохранил заявку
↓
NotificationService отправил админу уведомление
↓
handler показал клиенту подтверждение
```

Неправильный поток:

```text
handler сам проверяет товар
handler сам валидирует заявку
handler сам пишет в БД
handler сам отправляет админу текст
handler сам меняет статусы
```

Такой подход быстро приведёт к хаосу.

# 18. MVP Definition of Done

MVP можно считать готовым, если выполнены условия:

## Клиент

- пользователь запускает `/start`;
- видит главное меню;
- открывает каталог;
- выбирает категорию и подкатегорию;
- смотрит карточку товара;
- листает товары;
- оставляет заявку;
- видит свои заявки;
- может задать вопрос.

## Менеджер

- получает уведомление о новой заявке;
- видит детали заявки;
- может взять заявку в работу;
- может подтвердить заявку;
- может отклонить заявку;
- может завершить заявку;
- может ответить на вопрос клиента.

## Система

- заявки сохраняются в БД;
- товары берутся из БД;
- статусы заявок обновляются;
- клиент получает уведомления;
- действия менеджера логируются;
- seed-данные позволяют показать демо.

# 19. Главный риск проекта

Главный риск — превратить бота в набор разрозненных handlers.

Чтобы этого избежать:

```text
не писать бизнес-логику в handlers
не писать SQL прямо в handlers
не смешивать тексты, кнопки и логику
не делать огромные callback-функции
не пытаться сразу сделать все фичи
```

Правильный принцип:

```text
handler = входная точка
service = бизнес-логика
repository = база данных
text/keyboard = отображение
```

# 20. Главный стратегический вывод

RentalMarketBot нужно строить как **заявочную систему вокруг каталога**.

Не так:

```text
бот с каталогом товаров
```

А так:

```text
Telegram-система для арендного бизнеса:
каталог → заявка → менеджер → статус → ответ клиенту
```

Именно эта формула делает проект коммерчески понятным, портфельным и полезным для реального бизнеса.
