# План проверок RentalMarketBot

## Блок A — базовая техническая проверка

### T01. Проверка запуска проекта

Проверить, что проект импортируется, запускается без критических ошибок, 
все основные модули доступны.
- compileall
- import check
- запуск main в безопасном режиме

>[Результат: ]()
>- проект не запуститься без `TELEGRAM_TOKEN` и `POSTGRES_PASSWORD`
>- `build_app()` — не запустился (T01B)

---

### T02. Проверка зависимостей и конфигурации

Проверить .env, config.py, обязательные переменные, токен бота, DB URL, admin ids, DEBUG/PROD режим.

>[Результат: ]()Redis пока не используем.

---

### T03. Проверка Alembic / миграций

Проверить, что миграции применяются корректно:

- alembic current
- alembic heads
- alembic upgrade head

Без расхождения моделей и схемы БД.

>[Результат: ]()Все норм.

---

### T04. Проверка структуры проекта

Проверить, что структура соответствует архитектуре MVP:

handlers - services - repositories - schemas - models - states - status - middlewares - texts - keyboards

Без хаотичных зависимостей.

>[Результат: ]()Все норм.

---

### T05. Handler layer check

Проверить, что handlers:
- не ходят напрямую в repositories
- не содержат SQLAlchemy-запросы
- не принимают бизнес-решения
- не хранят большие тексты внутри себя

>[Результат:]() Тут бизнес-логика "залезла" в клав-ру через `status == ...Status` 
> 1. `get_admin_item_details_keyboard()` - `ItemStatus`
> 2. `get_admin_deal_details_keyboard()` - `RentalStatus`
> 3. `get_admin_support_ticket_keyboard()` - `SupportTicketStatus`
>
> Это повлияет на 
> - 3 функции `handlers/admin/admin_helpers/` (везде в функциях `send_or_edit`):
>   - `show_deal_card`
>   - `show_deal_contact`
>   - `show_support_ticket_card_or_not_found`
> - 5 функции `handlers/admin/item_moderation.py` (везде в функциях `send_or_edit`):
>   - `admin_items_view`
>   - `apply_item_status_action`
>   - `show_admin_item_card`
>   - `admin_items_save_quantity`
>   - `admin_items_save_price`
> - 1 функция `handlers/admin/support.py` (везде в функциях `send_or_edit`):
>   - `admin_support_close`

---

- проверяют права
- проверяют статусы
- проверяют доступность товара
- не знают Telegram UI
- не создают клавиатуры

### T07. Repository layer check

Проверить, что repositories:

- работают только с БД
- не знают про Telegram
- не знают про FSM
- не возвращают Pydantic DTO вместо ORM
- не содержат бизнес-решений

>[Результат:]() 
> Полностью убрал логику _[только открытые заявки]()_

---

### T08. Schema / DTO check

Проверить Pydantic-схемы:
- Create / Update / Out / Draft
- quantity
- price Decimal
- AwareDatetime
- from_attributes=True
- валидации min/max

>[Результат:]() 
> - support: `SupportMessageCreate` пока хз зачем
> - rental-repo: `RentalUpdate` заменен на `RentalStatusUpdate` в `_build_status_update()`

---

### T09. Model layer check

Проверить SQLAlchemy models:
- nullable
- foreign keys
- relationships
- indexes
- enum fields
- created_at / updated_at
- Decimal/Numeric для цен
- BigInteger для telegram_id
- Блок C — callback / keyboard / routing

>[Результат:]()
> - добавил enum_values
> - перевел все значения типа `REQUESTED` в lowercase `requested`

---

### T10. Проверка callback_data

Проверить, что каждая callback-кнопка имеет обработчик.

Примеры:
- item_details
- item_back
- rental_create
- rental_cancel
- admin_open_rental
- support_open
- category_open
- search_open_item

---

### T11. Проверка старых кнопок

Проверить защиту от stale callbacks:

- товар удалён
- товар скрыт
- заявка уже отменена
- заявка уже завершена
- старая кнопка из старого сообщения

---

### T12. Проверка клавиатур

Проверить reply/inline keyboards:

нет старой терминологии
нет битых callback
нет лишних кнопок
назад работает
главное меню работает
