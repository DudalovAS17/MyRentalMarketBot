# Error Handling for Callback Actions

## Пример блока обработки бизнес-ошибки

```python
    except (ValueError, PermissionError) as e:
        # Ошибка бизнес-логики: не тот статус / не владелец / не найдена сделка и т.п.
        await callback.answer(str(e), show_alert=True)  # (f"❌ {e}")
```

---

## Ошибки (у каждого — своё место обработки)

### 1) Парсинг `callback_data` (кнопка/ID) — «кнопка некорректная»
- Это `IndexError` / `ValueError`.
- Ловим в **хендлере**, потому что это входные данные.
- Обычно возникает там, где есть `callback.data.split(":")`.

**Примеры:**
- `IndexError` → кнопка без `:<id>` (устарела, баг, старое сообщение).
- `ValueError` → `<id>` не число.

**Причины:**
- Кто-то поменял формат `callback_data`, а старые сообщения остались в чате.
- Пользователь нажал кнопку из «старого» сообщения.
- Ты сам ошибся в генерации `callback_data`.
- `callback_data` обрезалась/склеилась (редко, но бывает при рефакторинге).

**Идеальный ответ:**
- `show_alert=True`, иначе пользователь не поймёт, почему ничего не произошло.
- Пример: `await callback.answer("Некорректная кнопка", show_alert=True)`.

### 2) Бизнес-логика
*(не тот статус / не тот пользователь / сделка не найдена / двойной клик / устаревшая кнопка)*

- Это то, что сервис должен уметь сообщать **контролируемо**.
- Ловим в **action-хендлере** (confirm/reject/cancel) и показываем пользователю.
- Это **бизнес-отказ**: система работает, но по правилам действие нельзя выполнить.

**Причины:**
- Статус уже изменился (двойной клик, другой участник успел изменить).
- Кнопка устарела (старое сообщение).
- Пользователь не тот (не владелец).
- Переход статусов запрещён (например, `REQUESTED → CONFIRMED` только владельцем).

**Идеальный ответ (UX):**
- Показать короткое понятное сообщение (часто `show_alert=True`).
- Перерисовать UI, чтобы пользователь увидел актуальный статус/кнопки.

---

### 3) UI-рендеринг (редактирование сообщения)
- Это `TelegramBadRequest`.
- Ловим в **render-функции**, потому что это «транспорт/интерфейс».

**Типичные ошибки `TelegramBadRequest`:**
- `message is not modified`
- `there is no text in the message to edit`
- `message can't be edited`
- `message to edit not found`

**Рекомендация:**
- Не ловить «вслепую» и всегда слать новое сообщение.
- `message is not modified` → можно проигнорировать.
- `can't edit` / `not found` / `no text` → отправить новое сообщение.

---

### 4) Неожиданная ошибка
- Это `Exception`.
- Ловим в **хендлере**, логируем, показываем «Попробуйте позже».

**Причины:**
- БД недоступна / таймаут / locked (SQLite).
- Ошибка запроса.
- Баг в коде сервиса.
- Исключение из зависимостей.

**Идеальный ответ:**
- `logger.exception(...)` для traceback.
- Пользователю: нейтральное сообщение «Ошибка. Попробуйте позже.»
- `return` — продолжать опасно (состояние неизвестно).
- Позже можно сузить `Exception` до `SQLAlchemyError/IntegrityError`, но принцип остаётся.

## Почему перерисовка нужна и при успехе, и при fail
Inline-кнопки живут в сообщении. Пользователь ожидает, что после нажатия:
- статус обновится, и/или
- кнопки изменятся.

Если не перерисовать, пользователь видит старые кнопки и снова нажимает → спам, путаница.

**Поэтому:**
- После успеха перерисовка обязательна.
- После бизнес-fail перерисовка очень полезна (покажет актуальный статус).
- После инфраструктурной ошибки перерисовку часто **НЕ** делают, потому что не ясно, что произошло
  (можно создать дополнительные ошибки).

---

## Сводка по 3 веткам (самая важная часть)

- **Некорректная кнопка (парсинг не удался)**
  → показать alert и выйти (нет смысла продолжать).

- **Инфраструктурная ошибка (сервис упал)**
  → лог + alert «попробуйте позже» + выйти (неизвестно, что реально произошло).

- **Бизнес-отказ (ok == False)**
  → alert «не удалось» + перерисовать детали (пользователь должен увидеть актуальное).

- **Успех (ok == True)**
  → короткий toast «Подтверждено» + перерисовать детали.

---

## Как использовать `callback.answer`

- Если нужно показать пользователю конкретный текст (успех/ошибка):
  `await callback.answer(text=..., show_alert=...)`
- Если не хочешь ничего показывать:
  `await callback.answer()` один раз в начале.

**Стандарт:**
- **action-хендлер**: `await callback.answer()` в конце (или один раз в нужной ветке с текстом).
- **show-details-хендлер**: `await callback.answer()` сразу, чтобы убрать «часики».
