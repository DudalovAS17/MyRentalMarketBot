# 🔎 MVP Search

## Цель
Дать пользователю быстрый способ найти нужное объявление и перейти к стандартному флоу аренды  
без навигации по категориям и лишних шагов.

Поиск — это **вход в публичный каталог**, а не отдельный сценарий аренды.

---

## Пользовательский сценарий

1. Пользователь нажимает кнопку **🔎 Поиск** в главном меню.
2. Бот переводит диалог в режим ожидания запроса.
3. Пользователь вводит текст (2–60 символов).
4. Бот показывает список найденных объявлений (страница 1).
5. Пользователь может:
   - открыть карточку объявления;
   - пролистать результаты;
   - изменить запрос;
   - вернуться в главное меню.
6. При открытии объявления используется **стандартная карточка** и **существующий флоу аренды**.

---

## Поведение поиска

### Что ищем
- Поиск выполняется по:
  - `Item.title`
  - `Item.description`
- Реализация: `ILIKE '%query%'` (без учёта регистра).

### Что показываем
Пользователю показываются **только публично доступные объявления**:
- `Item.status == ACTIVE`
- `Item.is_available == True` (если используется в публичном каталоге)

Поиск не показывает:
- PENDING
- HIDDEN
- REJECTED

---

## Ограничения запроса

- Минимум: 2 символа
- Максимум: 60 символов
- Пустой или некорректный запрос отклоняется с просьбой ввести новый

---

## Пагинация

- Размер страницы: **8 объявлений**
- Для определения `has_next` используется выборка `page_size + 1`
- Поддерживаются кнопки:
  - `⬅️ Пред`
  - `➡️ След`

---

## FSM-логика

Используется FSM для управления диалогом поиска.

### Состояния
- `waiting_for_query` — бот ожидает текст запроса
- `browsing` — пользователь просматривает результаты

### Инвариант FSM
Если бот просит ввести запрос — FSM **обязан** находиться в состоянии ожидания запроса.  
Для этого используется единая функция входа в поиск.

---

## UX-инварианты

- Пользователь никогда не попадает в “тупик”:
  - всегда есть кнопка “Новый запрос” или “Назад”
- Поиск не ломает другие сценарии
- Поиск не требует повторного ввода запроса при пагинации

---

## Архитектура

- SQL-запросы находятся в репозитории
- Бизнес-логика поиска — в сервисе
- Handlers отвечают только за UX и FSM
- Поиск переиспользует существующие методы каталога (без дублирования логики)

---

## Что НЕ входит в MVP Search

Осознанно не реализовано:

- фильтры (цена, категория, город)
- сортировка по релевантности или цене
- теги
- история поисков
- сохранённые запросы
- “похожие объявления”
- полнотекстовый / векторный поиск

---

## Критерии готовности (DoD)

MVP Search считается завершённым, если:

- [x] Пользователь может войти в поиск из меню
- [x] Бот принимает текстовый запрос
- [x] Отображаются только ACTIVE объявления
- [x] Работает пагинация
- [x] Можно открыть карточку объявления
- [x] FSM не рассинхронизируется
- [x] Поиск ведёт в стандартный флоу аренды
