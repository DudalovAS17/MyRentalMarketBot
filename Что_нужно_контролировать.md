# Codex

## Что стоит проверить/сделать перед тотальной унификацией

Ниже не “критично”, но это частые «забытые кусочки», которые помогают привести всё к единому стандарту быстрее и без конфликтов:

1. Валидация контракта между слоями
   - Убедиться, что handlers действительно не содержат бизнес‑логики, и используют сервисы только как «чёрный ящик».
   - Убедиться, что services не знают о Telegram/FSM объектах и не формируют UX‑ответы, а возвращают DTO/Schema наружу.
2. Жёсткое разграничение ORM и DTO
   - Архитектура прямо фиксиует правило: из сервисов наружу — только DTO/Pydantic, а ORM остаётся внутри репозиториев/сервисов. Это нужно выдержать до унификации, иначе потом будет больше переделок.
3. Единый подход к ошибкам
   - В архитектуре есть правило про глобальный error handler и запрет try/except в handlers для технических ошибок. Перед унификацией стоит убедиться, что код не расходится с этим правилом.
4. FSM‑данные → DTO
   - Архитектура допускает dict‑данные в FSM только на UX‑уровне, с обязательной нормализацией в DTO перед бизнес‑логикой. Это важно привести к единому виду до тотального рефакторинга, иначе появятся противоречия в стиле.

# GPT

1. Единый стандарт ошибок (ServiceError family) — ДО рефактора

   Сейчас у тебя местами ValueError, местами None, местами bool/int.
   Чтобы проект был “профессиональным”, нужно зафиксировать единый набор доменных ошибок:

   - ServiceError (база)
   - NotFoundError
   - ForbiddenError
   - ValidationError (бизнес-валидация)
   - ConflictError (инварианты: open rentals constraint, whitelist transitions)

   Закон:
   - repo исключения не нормализует (пробрасывает)
   - service выбрасывает только доменные ошибки
   - handlers ловят только доменные ошибки → UX

   Это супер важно, иначе “единый стиль” не удержится.

2. Соглашение по времени: UTC vs timezone-aware

   Ты в моделях уже используешь DateTime(timezone=True) и где-то datetime.utcnow().
   Нужно выбрать один стандарт:

   - либо всегда UTC и datetime.now(timezone.utc)
   - либо всегда naive UTC (хуже)

   Закон: сервисы пишут время одним способом, репо/модели не решают это.

3. Naming + DI ключи (data bag) — чтобы не было каши

   Ты активно используешь DI через data[...]. Надо зафиксировать:

   - как называются ключи (user_service, item_service, admin_ids, user)
   - что кладём в data и кто ответственен

   Пример закона:
   - ServicesMiddleware кладёт *_service
   - RegistrationMiddleware кладёт data["user"] (DTO или ORM — выбрать)
   - Admin middleware кладёт data["is_admin"]

   Иначе через месяц будет “data помойка”.

4. Явный порядок middleware (очень важно)

   Зафиксировать порядок, иначе баги:

   - GlobalErrorMiddleware
   - ServicesMiddleware
   - RegistrationCheckMiddleware
   - (опционально) AdminCheckMiddleware

   Это должно быть в ARCHITECTURE.md как правило.

5. Единый стиль индексов/constraints в ORM — уже почти есть

   Ты уже согласился: индексы только через Index(...) (или только index=True).
   Это надо вписать как закон и следовать при рефакторе моделей.

6. “ORM наружу не отдаём” — решить вопрос DictMixin/to_dict

   Ты сам показал to_dict() в ORM (Rental).
   Надо зафиксировать:

   - либо to_dict запрещён, наружу только DTO
   - либо to_dict разрешён только как debug helper (и не используется в сервисах)

   Я бы выбрал запрет (проф-стиль).
