# Services:

- Сервис бросает только бизнес-ошибки из utils.errors (не ValueError) через strict
- strict=True -> NotFoundError (это ошибка бизнес-сценария)
- Технические исключения не подавляются (DB/Network/etc.)
---

### **Service: НЕ логирует “пустой CRUD без смысла”**

CRUD-логи в сервисе делаются только если метод отражает бизнес-событие.
Если это просто CRUD без смысла — лог не нужен.

Вот тут важно:
- ❌ Не логируем “Example создан id=...”, если это просто техническая сущность без бизнес-события.
- ✅ логируем, если создание = бизнес-факт.

Пример:
- RentalService.create_rental() → это бизнес-событие “создана сделка” → лог MUST
- PhotoRepository.create() → это инфраструктура → лог не нужен
---

### ✅ Что именно логировать в сервисах (MUST):

    1) Переходы статусов (объявления / сделки / пользователи)
        кто инициатор (admin_id / user_id)
        что изменилось (old_status → new_status)
        сущность (item_id / rental_id / target_user_id)
        причина (если есть)

    2) Инварианты / Security / Guards
        попытка запретного действия (например скрыть item при open rentals)
        попытка бана админа/самобана
        попытка действия от BANNED (но у тебя это в middleware — там лог ок)

    3) Админские actions
        Админка — это всегда аудитопригодная зона. Там лог нужен почти всегда.

---

### **Service Strict-mode rule**
- **strict=False**: вернуть None, если не найдено.
- **strict=True**: если не найдено -> utils.errors (ошибка бизнес-сценария).


Репозитории не принимают бизнес-решений и могут возвращать None / False.
Сервисы обязаны интерпретировать такие результаты.
Для этого сервисные методы могут использовать параметр strict:
 - strict=False — допустимое отсутствие результата
 - strict=True — отсутствие результата считается ошибкой бизнес-сценария и приводит к исключению

Технические исключения (DB errors) не перехватываются strict и пробрасываются выше.
