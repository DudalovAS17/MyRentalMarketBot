# Архитектура проекта My-Rental-Bot

Этот документ фиксирует архитектурное ядро и единый стиль слоёв. Его цель — дать опорные правила, под которые можно постепенно приводить код.

## 1) Слои и поток данных

```
Handlers (UX/FSM/Router)
        ↓
Services (бизнес-логика)
        ↓
Repositories (SQLAlchemy/DB)
        ↓
DB models
```

### Базовый принцип
Слои не смешиваются.
Каждый слой:

- знает только про слой ниже, 
- не имеет прямого доступа к слоям выше или через них.

Нарушение этого правила считается архитектурной ошибкой, даже если код “работает”.

### Назначение слоёв (кратко)
- Handlers — взаимодействие с пользователем: UX, FSM, маршрутизация.
- Services — бизнес-правила и принятие решений.
- Repositories — доступ к данным и работа с БД.
- DB Models — описание структуры данных (ORM).

``` 
Разрешённые зависимости:
Handler  →  Service
Service  →  Repository
Repository → DB Models
```
---

## 2) Контракты слоёв (ядро)

### ✅ Handlers (UX / FSM / Router)
**Ответственность:** UX, FSM, маршрутизация, минимальная обработка входа.

Взаимодействие с пользователем и управление пользовательским сценарием.
Handlers отвечают только за интерфейс, но не за логику приложения.

**Разрешено:**
- работа с Message, CallbackQuery
- FSM:
    - установка / чтение / очистка состояний
    - хранение промежуточных данных сценария
- разбор входных данных (update/callback)
- формирование текста/клавиатуры
- вызов сервисов

**Запрещено:**
- прямые SQLAlchemy-запросы
- работа с ORM-моделями
- бизнес-логика (правила, проверки, статусные переходы)
- принятие решений уровня домена

**Принцип:** *"Handlers — тонкие и тупые". Они не знают почему что-то происходит, они знают только что показать.*

---

### ✅ Services (Business Logic)
**Ответственность:** бизнес-логика и правила приложения.

Service — это слой, где принимаются решения и проверяются инварианты.

**Разрешено:**
- принимать простые типы (int, str, bool) и DTO/Pydantic
- координировать несколько репозиториев
- проверять бизнес-инварианты (статусы, доступы, ограничения)
- реализовывать правила переходов состояний
- решать что делать в ответ на действие пользователя

**Запрещено:**
- Telegram-объекты (Message, CallbackQuery, FSMContext)
- FSM-объекты
- прямые SQL/SQLAlchemy запросы
- формирование пользовательского интерфейса

**Принцип:** *"Service = правила бизнеса". Сервис не знает как выглядит интерфейс, но знает что допустимо.*

---

### ✅ Repositories (SQLAlchemy / DB)
**Ответственность:** только доступ к БД и SQLAlchemy-логика.

Repository не содержит бизнес-смыслов.

**Разрешено:**
- любые SQLAlchemy-операции (query/filter/join/update/delete)
- транзакции (commit, rollback, refresh)
- управление сессией
- возврат ORM-моделей или примитивных значений (bool, int, exists)

**Запрещено:**
- бизнес-логика
- Telegram/FSM
- принятие решений уровня приложения
- формирование пользовательских сообщений

**Допускается:** 

- принимать DTO/Pydantic как *input* в репозитории (например, `ItemCreate`, `ItemUpdate`), если это упрощает работу сервиса. При этом репозиторий всегда возвращает ORM-объекты, а не DTO.

**Принцип:** *"Repository = чистый доступ к БД". Он знает как читать и писать, но не знает зачем.*


*правило текущего этапа (MVP): Transactions without UoW (project rule)*:
Проект не использует Unit of Work. Поэтому каждый write-метод репозитория обязан
выполнять rollback() при ошибке commit() и пробрасывать исключение выше.

---

### ✅ Schemas (Pydantic) — граница наружу
**Ответственность:** Schemas (Pydantic) определяют границу между бизнес-логикой и внешним миром (handlers, API, UI).

**Правило:** из сервисов наружу возвращаем только DTO/Pydantic, а не ORM.

ORM-модели:
- используются только внутри репозиториев и сервисов,
- не передаются в handlers напрямую.

Назначение: контракт между слоями и граница наружу.

**Обязательное:**
- В services принимаем Create/Update DTO

**Запрещено:**
- Использовать ORM в handlers
- Отдавать ORM наружу (в handlers/API)

**Рекомендуемый набор:**
- `XxxCreate`
- `XxxUpdate`
- `XxxOut`

---

### ✅ Models (ORM)

Назначение: описание структуры данных (DB models).

**Разрешено:**
- Только ORM-описание таблиц и связей
- Минимальная логика данных (computed поля, если допустимо)

**Запрещено:**
- Telegram/FSM/UX-зависимости
- Бизнес-логика и доменные решения
- Использование ORM вне repo/service

**Идентификаторы:**
- `db_user_id` — идентификатор пользователя в БД
- `telegram_user_id` — идентификатор пользователя в Telegram
- Никогда не смешивать эти типы ID

**Правило:**
- *не используем index=True*
- *Вместо него - __table_args__ = Index()*

---

### ✅ Middleware (глобальные политики приложения)

Назначение: глобальные политики уровня приложения (не бизнес-логика).

**Разрешено:**
- Глобальная обработка технических ошибок (DB/Network/etc.)
- Единое логирование stacktrace (один раз)
- Нейтральный ответ пользователю при технических сбоях
- Guards/политики (rate-limit, auth, banned-check)

**Запрещено:**
- Бизнес-логика и доменные решения
- Доступ к репозиториям/ORM напрямую
- Формирование бизнес-ответов (это зона handlers)

**Правило:**
- Технические ошибки перехватываются централизованно (global error handler)
- В handlers нет try/except для технических ошибок (только бизнес-ошибки)

---

**⚠️ Правило (проектное):**
Временные данные из FSM могут храниться и передаваться как dict
исключительно на уровне UX-потоков (Handlers).

Перед выполнением бизнес-логики сервис:
- обязан привести такие данные к DTO/Pydantic
- не должен опираться на “сырой” dict как на контракт

Использование dict допустимо как практичный компромисс для FSM/UX,
но не считается стандартным API сервисов.
---

## 3) Идентификаторы (строгое различение)

| Идентификатор | Что это |
|---|---|
| `db_user_id` | id пользователя в БД |
| `telegram_user_id` | id пользователя в Telegram |

**Правило:** никогда не путать. 

**Базовые правила**
- db_user_id никогда не используется для общения с Telegram API.
- telegram_user_id никогда не используется как первичный ключ БД.
- Все доменные сущности (Item.user_id, Rental.owner_id, Rental.renter_id)
должны явно использовать db_user_id, если не указано иное.

**Принцип:** *Идентификаторы — это семантика, а не просто числа.*

Если нужен пользователь по Telegram — сервис вызывает repo `get_by_telegram_id`.

## 4) Единый стиль API слоёв
Данный раздел фиксирует единый стиль сигнатур и ответственности методов.

### Репозитории
- `get_by_id`, `get_by_*`, `list_*`, `create`, `update`, `delete`

*Контракт*
- Возвращают ORM-объекты (или примитивы bool, int, None)
- принимают простые типы / dict / DTO (допускается) ???
- не возвращают Pydantic/DTO

### Сервисы
- Принимают простые типы (int, str, bool) и DTO
- Возвращают DTO/Pydantic
- Не знают про 
  - Telegram (Message, CallbackQuery)
  - FSM
  - клавиатуры / тексты сообщений

### Хэндлеры
- Получили ввод → вызвали сервис → ответили пользователю
- Форматирование сообщений выносить в helpers/formatters (по мере роста)


### from __future__ import annotations

`Отложенные аннотации типов  используются только 
в ORM-моделях и Pydantic-схемах - в models и schemas.`

`В handlers, services, repositories и middleware 
данный импорт запрещён, если нет явной необходимости`

## 6) Пример потока (словами)

1. Handler получает callback/update.
2. Handler выполняет минимальную обработку ввода и вызывает сервис: `ItemService.get_item_details(item_id)`.
3. Service выполняет бизнес-проверки (если нужны) и обращается к репозиторию: `ItemRepository.get_by_id(item_id)`.
4. Repository делает SQLAlchemy-запрос и возвращает ORM.
5. Service преобразует ORM → DTO/Pydantic и возвращает результат в handler.
6. Handler форматирует сообщение/клавиатуру и отвечает пользователю.

## 8) ⏱ Время в проекте

**Единый стандарт работы со временем во всём проекте.**

### Общее правило
- Во всех слоях домена и БД используется только **timezone-aware `datetime` в UTC**.
- Локальное время и форматирование допускаются **только на уровне UI / formatters**.

### Разрешено (и обязательно)
```python
from datetime import datetime, timezone
datetime.now(timezone.utc) # это aware и UTC
```

### Запрещено
- `datetime.now()` (→ naive)
- `datetime.utcnow()` (→ naive)
- хранение naive `datetime`
- сравнение `naive` и `aware` datetime

- ❌ Никаких timezone=False
- ❌ хранение локального времени в БД
- ❌ преобразование timezone в сервисах и ORM


## Закон времени проекта
- В БД и домене — только aware datetime в UTC
- Создание времени — только datetime.now(timezone.utc)
- ORM и сервисы не знают про локальное время
- Форматирование и локализация — только в helpers/formatters

### Обязательные требования

- В ORM и БД:
  - Все временные поля: `DateTime(timezone=True)`
  - в БД всегда хранится aware datetime
  - timezone = UTC
  - В БД хранится время в UTC (PostgreSQL: TIMESTAMP WITH TIME ZONE)
- Создание времени:
  - ✅ `datetime.now(timezone.utc)`
  - ❌ `datetime.now()`
  - ❌ `datetime.utcnow()`
- Сервисы и бизнес-логика:
  - работают **только** с aware `datetime`
  - не выполняют преобразований timezone
  - Сравнения времени выполняются только в UTC
- DTO / Pydantic (RentalOut и т.п.):
  - start_date: datetime | end_date: datetime
  - Используются `datetime` с timezone
  - сериализация (JSON) через `model_dump(mode="json")`. ➡️ Pydantic сам:
      - сохранит ISO-формат
      - не потеряет timezone
      - не сделает naive
- UI / formatters:
  - единственное место, где допустима:
    - локализация timezone
    - человекочитаемое форматирование
  - ТОЛЬКО здесь допустимо:
      - менять timezone
      - форматировать “для человека”
  - Пример:
    ```
    def format_dt(dt: datetime, tz: ZoneInfo) -> str:
    return dt.astimezone(tz).strftime("%d.%m.%Y %H:%M")
    ```
  - 📌 В Telegram:
    - можно брать timezone пользователя
    - или фиксировать, например, Europe/Moscow
    - или всегда писать “UTC” — это UX-решение


> Локальное время и формат — это задача отображения,  
> не доменной логики.


### В schemas:
- В Out-схемах: AwareDatetime ✅
- В Create/Update: лучше datetime (и нормализовать в сервисе) ✅
*Pydantic v2 и так примет ISO-строку и распарсит в datetime, но AwareDatetime будет требовать tzinfo.*
